# 后台管理复用框架协作规范

> 本文件是本项目的人机协作与工程治理准则，适用于 Claude、Codex、Gemini 等所有代码代理和人工开发者。  
> 项目定位：基于 Bun + Hono + Drizzle ORM 的轻量后台管理复用框架，目标是在多个后台项目之间复用用户、认证、RBAC、系统配置、日志、安全中间件、数据库适配等通用能力，降低重复开发成本。  
> 核心原则：小而美不是砍掉能力，而是以清晰模块、低接入成本、高复用杠杆实现可裁剪、可维护、可落地的后台瑞士军刀。

## 1. 项目铁律

1. **代码变更必须入库**：每次代码变更完成并验证后，必须提交到代码仓库；commit message 使用中文，并说明为什么改、验证了什么、还有哪些未验证风险。
2. **代码与文档同步**：所有代码修改都必须同步更新相关文档；如果修改的是行为、接口、配置、数据库、部署、测试方式，必须在 `docs/` 中留下说明。
3. **先全局分析再动手**：开发前必须理解项目定位、模块关系、调用链、数据流和复用目标，不能只看局部不停打补丁。
4. **小而美，反对过度设计**：不做大而全系统，不为假想需求堆功能；但后台管理高频通用能力要沉淀为可复用模块，而不是每个项目重复写。
5. **闭环交付**：每次改动必须形成“分析 → 方案 → 实现 → 文档 → 测试 → 复盘”的闭环，不能只改代码不验证。
6. **核心优先**：开发围绕认证、用户、RBAC、系统配置、资源 CRUD、数据库适配等核心后台复用能力展开；非核心细节写入 `docs/todolist.md`，不要阻塞主线交付。
7. **落地优先**：所有设计必须站在真实后台管理场景考虑，避免假大空、炫技式抽象和无法维护的方案。
8. **最佳实践优先**：技术选型、算法、权限模型、安全策略、数据库访问方式优先采用行业成熟方案；除非有明确收益，不重复造轮子。
9. **工具提效**：优先使用 `rtk`、MCP、skills、脚本和自动化测试提高效率，避免人工重复排查。
10. **脚本沉淀**：构建、测试、调试脚本应正式化、可复用、可追踪；严禁反复写一次性临时脚本又删除，导致经验无法沉淀。
11. **关键注释**：核心代码、关键变量、业务常量、权限规则、复杂流程必须补充清晰中文注释；注释解释“为什么”和“业务约束”，不要重复代码表面含义。
12. **ORM 优先**：能使用 Drizzle ORM 的场景禁止手写原始 SQL；确需原始 SQL 时必须说明原因、做好参数绑定、防 SQL 注入，并记录到文档。

## 2. 架构原则

### 2.1 复用框架定位

本项目不是单次业务项目，也不是纯 demo，而是后台管理项目的复用底座。新增能力必须判断其属于哪一层：

- **核心内核**：应用创建、配置、数据库、错误处理、日志、安全中间件、Auth、User、RBAC、System Config、资源 CRUD 规范。
- **标准模块**：菜单、字典、审计日志、文件上传、导入导出、指标、接口文档。
- **业务扩展**：具体项目的订单、产品、客户、设备、审批等业务模块。

核心内核必须稳定、克制、可测试；标准模块必须可开关；业务扩展不能污染核心内核。

### 2.2 模块边界

新增或重构模块时，必须明确：

- 该模块解决什么真实后台问题。
- 对外 Interface 是什么。
- 是否存在可替换 Adapter。
- 与 Auth、RBAC、数据库、配置、日志的关系。
- 如何被业务项目复用。

禁止把所有逻辑堆进 `main.js`、controller 或通用工具函数中。优先保持 `routes → controller/service → repository/model → schema` 的清晰结构。

### 2.3 不做“为了 lite 的删减”

本项目的小而美强调：

- 接入成本低。
- 模块开关清晰。
- 默认路径好用。
- 通用能力可复用。
- 业务扩展有规范。

用户、认证、RBAC、系统配置、安全中间件不是负担，而是后台管理复用的核心资产。优化方向是模块化、标准化、可裁剪，不是简单删除。

## 3. 文档规范

### 3.1 docs 目录是知识库

所有项目文档、分析文档、踩坑记录、设计决策、测试说明、接口约定都必须记录在根目录 `docs/` 下。

文档命名规范：

```text
序号_中文文件名_日期.md
```

要求：

- 序号使用三位数字，如 `001`、`002`。
- 文件名使用中文，不含空格。
- 日期使用 `YYYY-MM-DD`。
- 示例：`001_权限模型设计_2026-07-05.md`。

已有历史文档如未按规范命名，后续整理时再迁移，不要在无关任务中大规模改名。

### 3.2 踩坑记录模板

发现问题、规范缺陷、流程问题、工具链坑点时，必须写入 `docs/`。踩坑记录至少包含：

```markdown
# 001_问题标题_YYYY-MM-DD

## 背景

## 现象

## 影响范围

## 原因分析

## 排查过程

## 解决方案

## 验证方式

## 后续建议
```

### 3.3 必须同步的文档类型

- 接口变更：更新接口说明、curl 示例或前端契约。
- 数据库变更：更新 schema、初始化、迁移与兼容性说明。
- 权限变更：更新权限命名、角色授权、菜单/按钮权限说明。
- 配置变更：更新配置项、默认值、安全注意事项。
- 测试变更：更新测试命令、测试数据和覆盖范围。
- 架构变更：更新设计说明或 ADR。

## 4. 开发流程

### 4.1 开发前

1. 阅读相关代码、README、`docs/` 中已有分析和约定。
2. 明确本次需求属于核心内核、标准模块还是业务扩展。
3. 识别影响范围：路由、模型、数据库、权限、配置、测试、文档。
4. 给出顶层思路，避免只修表面现象。

### 4.2 开发中

1. 优先改根因，避免堆补丁。
2. 复用已有工具、模型、中间件和约定。
3. 不引入新依赖，除非已有方案无法低成本满足需求。
4. 新增脚本必须可复用、可提交、可文档化。
5. 涉及权限、认证、数据库、配置、安全时必须格外保守。

### 4.3 开发后

1. 运行最小相关测试，再按风险运行更大范围测试。
2. 更新 `docs/` 相关文档。
3. 检查是否有未实现接口、调试代码、临时脚本、危险默认配置。
4. 用中文 commit 提交代码。
5. 最终说明必须包含：改动文件、解决的问题、测试证据、文档位置、剩余风险。

## 5. 测试规范

### 5.1 测试优先级

后台复用框架必须重点覆盖：

- Auth：登录、刷新 token、无效 token、过期 token、登出。
- User：创建、更新、禁用、软删除、分页、搜索。
- RBAC：角色授权、权限检查、直接授权、拒绝授权、过期授权。
- Resource CRUD：列表、详情、新增、编辑、删除、搜索、排序、分页。
- Database Adapter：SQLite 必测，MySQL 至少做 schema/初始化 smoke test。
- Middleware：错误响应、请求大小、CORS、安全头、限流。

### 5.2 测试脚本沉淀

- 能写正式测试就不要写临时测试。
- 需要调试脚本时，放到 `scripts/` 或 `tests/`，命名清晰，并补充说明。
- 不要用“写了删、删了写”的方式反复试错。
- 测试数据初始化必须可重复执行。

### 5.3 验证标准

每次提交前至少确认：

- 相关测试通过。
- 文档已同步。
- 无未实现接口被误暴露。
- 无敏感信息硬编码。
- 数据库访问优先使用 ORM。
- 变更不会破坏已有后台复用路径。

## 6. 数据库与 ORM 规范

1. 默认使用 Drizzle ORM 表达查询和写入。
2. 禁止拼接 SQL 字符串。
3. 排序、搜索、过滤字段必须使用白名单。
4. 分页必须下推到数据库，避免先查全量再内存切片。
5. SQLite 与 MySQL 的 schema 能力应保持一致；暂不一致时必须在文档中明确限制。
6. 数据库初始化、索引、种子数据必须可重复执行。
7. 权限、角色、系统配置等基础数据必须有明确 seed 策略。

## 7. 权限与安全规范

1. 权限命名统一采用 `resource:action`，如 `users:list`、`roles:update`。
2. 常用动作优先使用：`list`、`read`、`create`、`update`、`delete`、`export`、`import`、`approve`。
3. 默认角色建议包括：`super_admin`、`admin`、`operator`、`viewer`。
4. 新增后台资源时，必须同步定义权限点。
5. 认证、授权、菜单、按钮权限之间的关系必须文档化。
6. JWT secret、数据库密码、API key 等敏感信息禁止硬编码。
7. CORS、CSP、限流、安全头默认值必须安全、可解释、可配置。

## 8. 提交规范

### 8.1 commit 语言

commit message 必须使用中文。

### 8.2 推荐格式

```text
说明本次变更的目的，而不只是描述改了什么

补充背景、约束、方案取舍和影响范围。

约束: <影响决策的外部条件>
取舍: <拒绝的方案> | <拒绝原因>
信心: <低|中|高>
风险: <窄|中|宽>
验证: <运行过的测试或人工验证>
未验证: <已知未覆盖的风险>
```

示例：

```text
统一后台权限命名以支撑多项目复用

用户、角色、菜单和按钮权限此前缺少统一命名约定，导致不同项目接入时容易分叉。本次补充 resource:action 规范，并为默认角色预留初始化策略。

约束: 需要兼容现有 roles 与 permissions 表
取舍: 引入复杂策略引擎 | 当前项目暂不需要 ABAC 复杂度
信心: 高
风险: 窄
验证: bun test
未验证: MySQL 初始化脚本
```

## 9. 代理协作要求

1. 开始任务前先看本文件和相关 `docs/`。
2. 发现规则冲突时，以用户最新明确目标和本文件项目定位为准；仍冲突则说明原因。
3. 简单任务直接执行，复杂任务先列短计划。
4. 修改前检查工作区状态，避免覆盖他人改动。
5. 不要为了完成任务而绕开测试、文档或安全要求。
6. 遇到不确定问题，优先通过读代码、查文档、跑测试验证，不要猜。
7. 使用工具提高效率，但工具输出必须经过判断，不能机械照搬。
