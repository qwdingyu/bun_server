# 后台前端集成契约

> 目标：在不集成 Logto 的前提下，为 Vue/React/任意管理后台前端提供一套低成本、可复用、可替换的本地认证与 RBAC 集成约定。本文只定义当前 Local JWT 方案，不引入 OIDC、BFF、多租户等重型能力。

## 1. 适用场景

适合：

- 多个后台管理项目复用同一套登录、用户、权限基础设施。
- 前后端分离后台，例如 Vue Admin、React Admin、内部运营平台。
- 默认 SQLite 快速落地，后续可迁移到 MySQL。
- 暂不接入第三方身份服务，但希望未来可替换到 Logto 等 Provider。

不适合：

- 面向公网的大型统一身份平台。
- 需要复杂组织、多租户、SSO、SAML/OIDC federation 的项目。
- 强合规场景下需要 HttpOnly Cookie/BFF 的认证体系。

## 2. 前端认证状态模型

前端只需要维护四类状态：

```text
auth.accessToken          当前访问令牌
auth.refreshToken         当前刷新令牌
auth.user                 当前用户安全字段
auth.permissions          当前用户权限名数组，用于路由和按钮判断
auth.menus                当前用户可见菜单树和按钮清单
```

当前已提供 `GET /api/users/me/permissions` 与 `GET /api/users/me/menus`。前端登录后应拉取一次权限和菜单；权限变更、角色变更或重新登录后重新拉取。

推荐存储：

| 数据 | 推荐位置 | 原因 |
| --- | --- | --- |
| `accessToken` | 内存 + 可选 `sessionStorage` | 生命周期短，降低长期泄露风险 |
| `refreshToken` | `localStorage` 或 `sessionStorage` | 当前纯 API 方案无法使用 HttpOnly Cookie，只能由前端持有 |
| `user` | 状态管理 + 可选本地缓存 | 页面展示需要 |
| `permissions` | 状态管理 | 权限变更后应重新拉取 |
| `menus` | 状态管理 | 用于渲染侧边栏、路由入口和按钮清单 |

最小实现建议：

- 内部后台优先用 `sessionStorage`，关闭标签页即退出。
- 需要“记住登录”时再使用 `localStorage` 保存 refresh token。
- 不要把 token 放到 URL、日志、埋点、错误上报中。

## 3. 登录流程

```text
1. 用户提交用户名/邮箱 + 密码。
2. 前端调用 POST /api/users/auth/login。
3. 后端返回 user、accessToken、refreshToken。
4. 前端保存 token，并写入 auth store。
5. 前端调用 `GET /api/users/me` 或直接使用登录响应 user 初始化用户状态。
6. 前端调用 `GET /api/users/me/permissions` 和 `GET /api/users/me/menus` 初始化权限与菜单。
7. 跳转到后台首页。
```

请求：

```http
POST /api/users/auth/login
Content-Type: application/json

{
  "identifier": "admin",
  "password": "password123"
}
```

前端处理规则：

- `200 + success=true`：写入登录态。
- `401`：提示账号或密码错误。
- `403`：提示账号被禁用或无权限。
- `429`：提示请求过于频繁。
- 其他 `5xx`：提示系统繁忙，不要暴露后端堆栈。

## 4. 请求拦截器契约

所有受保护 API 都需要：

```text
Authorization: Bearer <accessToken>
```

前端请求拦截器职责：

1. 从 auth store 读取 `accessToken`。
2. 有 token 时追加 `Authorization`。
3. 不在无认证接口强行追加 token，例如登录、注册、刷新。
4. 不修改业务请求体结构。

伪代码：

```js
request.use((config) => {
  if (auth.accessToken && !config.skipAuth) {
    config.headers.Authorization = `Bearer ${auth.accessToken}`
  }
  return config
})
```

## 5. 刷新令牌流程

刷新触发条件：

- API 返回 `401` 且错误码为 token 失效类。
- 或前端按 `expiresIn` 预判 access token 即将过期。

刷新流程：

```text
1. 暂停当前失败请求。
2. 如果已有刷新请求在进行，等待同一个 Promise。
3. 调用 POST /api/users/auth/refresh，传入 refreshToken。
4. 刷新成功后更新 accessToken 和 refreshToken。
5. 重放原失败请求。
6. 刷新失败则清空登录态并跳转登录页。
```

并发控制要求：

- 同一时间只允许一个 refresh 请求。
- 多个并发 401 等待同一个刷新结果。
- 旧 refresh token 刷新成功后立即失效，前端必须使用后端返回的新 refresh token 覆盖旧值。

伪代码：

```js
let refreshPromise = null

async function refreshTokensOnce() {
  if (!refreshPromise) {
    refreshPromise = api.post('/api/users/auth/refresh', {
      refreshToken: auth.refreshToken
    }).finally(() => {
      refreshPromise = null
    })
  }
  return refreshPromise
}
```

## 6. 登出流程

前端主动登出：

```text
1. 调用 POST /api/users/auth/logout，携带 accessToken。
2. 请求体传入 refreshToken。
3. 无论接口成功或失败，都清空本地登录态。
4. 跳转登录页。
```

请求：

```http
POST /api/users/auth/logout
Authorization: Bearer <accessToken>
Content-Type: application/json

{
  "refreshToken": "..."
}
```

说明：

- `refreshToken` 建议必传，后端才能撤销服务端会话。
- 如果 access token 已失效，前端也应清空本地状态，不要无限重试。

## 7. 路由守卫契约

前端路由元信息建议：

```js
{
  path: '/users',
  meta: {
    requiresAuth: true,
    roles: ['admin', 'super_admin'],
    permissions: ['users:list']
  }
}
```

守卫顺序：

```text
1. 不需要登录的页面直接通过。
2. 需要登录但无 accessToken：跳转登录页。
3. 有 accessToken 但无 user：调用 /api/users/me 恢复用户态。
4. 如果配置 roles：校验用户角色。
5. 如果配置 permissions：校验权限数组。
6. 校验失败跳转 403 页面。
```

当前第一阶段注意：

- 后端 token payload 中有 `role` 字段，但角色/RBAC 真正来源应以后端数据库为准。
- 当前 `/api/users/me/permissions` 返回有效权限数组和权限名数组；`/api/users/me/menus` 返回按权限过滤后的菜单和按钮。

## 8. 菜单与按钮权限约定

当前菜单数据结构：

```json
{
  "key": "users",
  "title": "用户管理",
  "path": "/users",
  "icon": "users",
  "permission": "users:read",
  "buttons": [
    {
      "key": "create",
      "title": "新增用户",
      "permission": "users:create"
    }
  ]
}
```

前端处理规则：

- 页面路由用 `permission` 控制是否可进入。
- 按钮用 `permission` 控制是否显示或禁用。
- 后端仍必须校验接口权限，前端权限只提升体验，不能作为安全边界。

## 9. 统一错误处理

错误响应格式见 `010_后台接口契约_2026-07-05.md`。

前端建议处理表：

| HTTP 状态 | 前端动作 |
| --- | --- |
| 400 | 表单或请求参数错误，展示后端 message |
| 401 | 尝试刷新 token；刷新失败则跳登录页 |
| 403 | 跳转 403 页面或显示无权限 |
| 404 | 跳转 404 页面或显示资源不存在 |
| 409 | 展示冲突信息，例如用户名已存在 |
| 422 | 表单字段级错误 |
| 429 | 展示限流提示，短时间禁用提交按钮 |
| 500 | 展示系统繁忙，记录 requestId |

前端不要依赖中文 message 做逻辑判断，应优先依赖 `error.code`。

## 10. 安全要求

必须做：

- HTTPS 部署。
- token 不写入 URL。
- token 不写入日志、埋点、Sentry breadcrumb。
- refresh token 刷新后立即覆盖旧值。
- 退出登录清空 accessToken、refreshToken、user、permissions。
- 生产环境配置明确 CORS origin，不使用 `* + credentials`。

暂不做：

- 自研 OIDC。
- 自研 SSO。
- 多端设备管理页面。
- 复杂风控和设备指纹。

可后续增强：

- HttpOnly Cookie + BFF 模式。
- 登录失败 IP/用户双维度限流。
- 当前会话列表与踢下线。
- 菜单元数据热更新或数据库化配置。
- 审计日志查询后台。

## 11. 前端最小落地清单

第一天可落地：

1. 登录页。
2. auth store。
3. request/response 拦截器。
4. refresh token 单飞机制。
5. 路由守卫。
6. 403/404 页面。
7. 当前用户展示。

不要第一天做：

- 复杂权限可视化配置页。
- 组织/租户切换。
- SSO Provider 抽象 UI。
- 自定义策略语言。

## 12. 与未来 Logto 的兼容边界

如果未来接入 Logto，前端改动应限制在：

- 登录页跳转方式。
- token 获取方式。
- auth store 初始化来源。
- refresh/logout SDK 调用方式。

不应影响：

- 业务 API 的 `Authorization: Bearer` 约定。
- 后端 `AuthContext`。
- 业务页面权限判断结构。
- 菜单/按钮权限数据结构与项目侧扩展菜单。
