# npm 发布形态与复用策略评估

日期：2026-07-05  
评估对象：`bun_server` 后台管理脚手架  
参考对象：`/Users/dingyuwang/0-X/2-node/publish-toolkit`、`/Users/dingyuwang/0-X/2-node/cf-core/.github/workflows/publish.yml`

## 1. 结论先行

当前不建议把本项目直接当作普通 npm 依赖包发布，然后让其他项目通过 `npm install bun-server-framework` 来使用。

更合理的路线是分两步：

1. **短期：不要发布为普通库，优先作为 Git 模板/脚手架源码复用。**
2. **中期：如果确实要走 npm，发布成 CLI 脚手架包，而不是运行时依赖库。**

原因：本项目本质是“后台项目起始模板”，包含应用入口、配置、数据库初始化、脚本、文档、测试、默认 seed、服务启动方式。它不是像 `@usethink/cf-core` 那样的可 import 运行时内核库。直接作为普通 npm 包安装，会让使用者拿到 `node_modules/bun-server-framework`，但业务项目真正需要的是一份可修改、可提交、可长期演进的源码工程。

一句话建议：**当前适合先做 template/scaffold，不适合直接做 runtime dependency；如果发布 npm，也应发布 `create-*` 或带 `bin` 的脚手架 CLI。**

## 3. 参考 publish-toolkit 的能力

`/Users/dingyuwang/0-X/2-node/publish-toolkit` 已是 npm 包：

```json
{
  "name": "@usethink/publish-toolkit",
  "version": "0.1.15",
  "bin": {
    "publish-toolkit": "dist/cli.js"
  }
}
```

它提供：

- `publish-toolkit publish`
- `publish-toolkit obfuscate`

发布命令能力包括：

- 检查 `package.json`。
- 拒绝 `private: true` 的包。
- 检查 git 工作区是否干净。
- 检查版本号是否已发布。
- 如存在 `build` 脚本，正式发布时执行构建。
- 写入临时 `.npmrc` 使用 `NPM_TOKEN`。
- 执行 `npm publish` 或 `pnpm publish`。
- 支持 `--dry-run`、`--tag`、`--access`、`--registry`。

重要限制：

- `publish-toolkit` 解决的是“如何安全发布一个已经准备好的 npm 包”。
- 它不负责判断当前项目是否适合作为 npm 包。
- 它不会自动把一个应用工程改造成 CLI 脚手架。

因此，本项目可以复用 `publish-toolkit` 的发布能力，但必须先设计清楚发布形态。

## 4. 参考 cf-core workflow 的适用边界

`/Users/dingyuwang/0-X/2-node/cf-core/.github/workflows/publish.yml` 的核心流程是：

```text
1. push tag v* 或 workflow_dispatch 触发。
2. setup-node@v4，Node 22。
3. corepack 启用 pnpm。
4. npm ci 安装依赖。
5. 打印 package 信息。
6. npm run type-check。
7. npm test。
8. npm run build。
9. pnpm exec publish-toolkit publish。
```

该 workflow 适合 `cf-core`，因为 `cf-core` 是标准库包：

- 有 `main`。
- 有 `types`。
- 有 `exports`。
- 有 `files`。
- 有 TypeScript 构建。
- 有 `type-check`。
- 有 `prepack`。
- 有 `@usethink/publish-toolkit` devDependency。

当前 `bun_server` 不满足这些前提：

- 当前是 Bun 应用脚手架，不是 TypeScript 库。
- 当前没有 `type-check` 脚本。
- 当前 `build` 是编译服务端二进制，不是生成可 import 的 `dist` 包。
- 当前没有 `files` 白名单，`npm pack` 会包含很多模板维护文件。
- 当前没有 `bin`，用户安装后没有 `create` 命令。
- 当前没有 `exports`，用户也没有明确可 import 的稳定 API。

所以不能直接照搬 `cf-core` 的 workflow；可以借鉴触发方式、NPM_TOKEN、dry-run、publish-toolkit 调用，但发布前置步骤必须按本项目形态重写。

## 5. 三种可选发布形态

### 5.1 方案 A：不发 npm，作为 Git 模板复用

使用方式：

```text
1. 从 Git 仓库复制或使用 GitHub Template。
2. 修改项目名、配置、README、默认管理员和 JWT_SECRET。
3. 按 docs/015 和 docs/019 新增业务资源。
```

优点：

- 最符合当前项目源码形态。
- 使用者拿到完整可修改工程。
- 不需要先做 CLI、打包白名单、模板复制逻辑。
- 对“小而美、快速落地”最友好。

缺点：

- 多项目同步脚手架升级需要人工 cherry-pick 或文档指导。
- 不能用 `npm create` 一键生成项目。

适用阶段：当前阶段。

结论：**短期推荐。**

### 5.2 方案 B：发布为普通 npm 依赖库

使用方式：

```bash
npm install bun-server-framework
```

理论上可以导入：

```js
import { createApp } from 'bun-server-framework'
```

但当前项目没有设计这种稳定导出契约。

问题：

- 业务项目无法方便修改内置 schema、路由、控制器、seed。
- 数据库初始化、配置文件、脚本、docs 都在 `node_modules` 中，不符合后台项目二次开发习惯。
- 业务项目若 fork 内部代码，会绕过 npm 依赖意义。
- 当前没有 `exports` 和稳定 public API。
- 如果把整个 app 作为依赖，后续版本升级会牵涉业务覆盖和数据迁移，风险高。

适用条件：

只有当本项目拆出稳定内核库，例如：

- `@usethink/bun-admin-core`
- 只导出 Hono middleware、auth provider、RBAC helper、logger、config、db helper
- 不包含业务项目模板和默认应用入口

才适合普通 npm library 形态。

结论：**当前不推荐。**

### 5.3 方案 C：发布为 npm CLI 脚手架包

使用方式可以设计为：

```bash
npm create bun-server-framework my-admin
```

或：

```bash
npx bun-server-framework create my-admin
```

包形态：

```json
{
  "name": "create-bun-server-framework",
  "bin": {
    "create-bun-server-framework": "dist/cli.js"
  },
  "files": [
    "dist",
    "templates",
    "README.md"
  ]
}
```

CLI 职责：

- 复制模板到目标目录。
- 替换 `package.json` name。
- 可选选择 SQLite/MySQL。
- 生成 `.env.example`。
- 提醒修改 `JWT_SECRET` 和默认管理员。
- 可选执行 `bun install`。
- 输出下一步命令。

优点：

- npmjs 发布有意义。
- 使用者拿到的是源码工程，而不是运行时依赖。
- 可以保留脚手架“快速落地”的核心价值。
- 后续可以迭代模板版本。

缺点：

- 需要新增 CLI 和模板目录。
- 需要处理模板文件白名单。
- 需要设计版本升级策略。
- 需要避免把本仓库运行时数据、日志、内部 agent 文档发布进去。

适用阶段：当本脚手架在 2-3 个项目复用后，模板边界稳定时。

结论：**中期推荐。**

## 6. 推荐路线图

### 6.1 当前阶段：先不发布 npm

当前最稳妥做法：

1. 继续用 Git 复制或 GitHub Template 方式复用。
2. 完善 docs/020 的使用说明和踩坑指南。
3. 在真实项目中验证新增资源模板。
4. 记录哪些文件每次复制后都要改。
5. 记录哪些模块每个项目都会删或改。

验收标准：

- 至少 2 个后台项目从本脚手架成功落地。
- 新增资源流程没有明显返工。
- 默认认证/RBAC 能满足大多数后台项目。
- 常见裁剪项稳定。

### 6.2 准备发布 CLI 前需要完成

如果决定发布 npm CLI，需要先完成：

1. 新增 `templates/default/`，把可复制工程放进去。
2. 新增 CLI 入口文件，或单独维护一个 TypeScript CLI 包。
3. `package.json` 增加 `bin`。
4. `package.json` 增加 `files` 白名单，只发布 CLI、模板、README、必要 docs。
5. 增加 `.npmignore` 或 `files`，避免发布：
   - `.git/`
   - `node_modules/`
   - `data/`
   - `logs/`
   - `.DS_Store`
   - 本地测试数据库
   - 不适合公开的内部协作文件
6. 增加 `dry-run` 或本地生成测试。
7. 增加 workflow。
8. 用 `npm pack --dry-run --json` 审查最终包内容。

### 6.3 发布 workflow 建议

不能直接照搬 `cf-core`，应改成 Bun/CLI 脚手架适配版。

建议流程：

```text
1. push tag v* 或 workflow_dispatch 触发。
2. actions/checkout@v4。
3. setup-node@v4，registry-url 指向 npmjs。
4. oven-sh/setup-bun 安装 Bun。
5. npm install 或 bun install，取决于最终 lockfile 策略。
6. bun run test:all。
7. 如有 CLI 构建，则执行 build。
8. npm pack --dry-run --json，审查包内容。
9. publish-toolkit publish --dry-run 或正式发布。
```

如果沿用 `publish-toolkit`：

```yaml
- name: Publish to npm
  run: |
    if [ "${{ github.event_name }}" = "workflow_dispatch" ] && [ "${{ inputs.dry_run }}" = "true" ]; then
      npx @usethink/publish-toolkit publish --dry-run --registry=https://registry.npmjs.org/ --verbose
    else
      npx @usethink/publish-toolkit publish --registry=https://registry.npmjs.org/ --verbose
    fi
  env:
    NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

也可以把 `@usethink/publish-toolkit` 加入 `devDependencies` 后使用：

```text
bunx publish-toolkit publish --dry-run
```

或：

```text
npx @usethink/publish-toolkit publish --dry-run
```

具体选择取决于最终是否希望仓库依赖中固定 publish-toolkit 版本。

## 7. 如果坚持现在发布，会有哪些风险

### 7.1 使用方式不自然

用户执行：

```bash
npm install bun-server-framework
```

得到的是依赖包，而不是新项目目录。后台项目通常需要修改源码、配置、schema、seed 和文档，因此 `npm install` 不符合主要使用场景。

### 7.2 包内容过宽

当前 `npm pack --dry-run` 会打入 `CLAUDE.md`、`git_push.sh`、全部测试和全部 docs。虽然 `data/`、`logs/`、`node_modules/` 被 `.gitignore` 排除，没有进入 pack，但当前包内容仍不是精心设计的发布物。

### 7.3 构建语义不匹配

当前 `bun run build` 会生成服务端二进制：

```text
bun build --compile --minify --outfile dist/server src/main.js
```

这不是 library 的 JS 构建，也不是 CLI 的模板复制构建。照搬 `cf-core` 的 `npm run build` 再发布，不能证明包可被正确消费。

### 7.4 版本升级难

如果其他项目把它当依赖，一旦脚手架内部 schema、seed、路由变化，业务项目如何升级会很复杂。源码模板模式反而更透明。

## 8. 建议的 package 命名

如果未来做 CLI，建议优先考虑 scoped package，避免通用名称冲突：

- `@usethink/create-bun-admin`
- `@usethink/create-bun-server`
- `@usethink/bun-admin-scaffold`

如果坚持非 scoped：

- `create-bun-admin-scaffold`
- `create-bun-server-framework`

不建议把 CLI 和 runtime library 混用同一个包名。`bun-server-framework` 更像项目名，不够明确表达 CLI 用法。

## 9. 决策建议

| 选项 | 当前是否建议 | 原因 |
| --- | --- | --- |
| 直接发布当前包到 npm | 不建议 | 当前是应用脚手架源码，不是 library/CLI |
| GitHub Template / Git 复制 | 建议 | 最符合当前形态，成本最低 |
| 发布 CLI 脚手架包 | 中期建议 | 需要先抽模板目录和 CLI |
| 拆 runtime core 包 | 暂缓 | 可能过度抽象，需多个项目验证重复点 |
| 复用 publish-toolkit | 建议 | 适合作为最终发布动作，但不是形态设计工具 |
| 照搬 cf-core workflow | 不建议 | cf-core 是 TS library，本项目是 Bun scaffold |

## 10. 最终结论

当前项目的复用目标是快速搭建后台管理系统，核心价值在“复制后可改、可跑、可测、可裁剪”。因此，短期最优解不是马上发布到 npmjs，而是继续作为模板工程复用。

如果后续希望通过 npmjs 分发，推荐把发布目标改造成 CLI 脚手架包：用户通过 npm 执行 create 命令，生成一份完整源码项目。届时可以复用 `@usethink/publish-toolkit` 和 `cf-core` workflow 的发布思想，但 workflow 必须按 Bun 脚手架项目重新设计。

一句话：**不要把脚手架发布成别人 import 的依赖；要发布，也应发布成能生成项目的 create CLI。**
