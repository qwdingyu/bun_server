# 抽象重构必要性评估

> 目标：回答“是否有需要抽象、重构的地方”。本文遵循小而美后台脚手架原则：不为了重构而重构，只在能提升复用、降低重复开发、增强测试和维护 locality 时才建议动代码。

## 1. 评估原则

本次评估使用以下判断标准：

- **复用收益**：是否会在多个后台项目重复出现。
- **接口深度**：是否能用小 interface 隐藏较多 implementation。
- **locality**：修改是否能集中在一个地方，而不是散落多个控制器和路由。
- **测试收益**：是否能让测试更聚焦于稳定 seam。
- **风险成本**：是否会破坏现有接口契约、增加复杂度或引入新依赖。
- **删除测试**：如果删除该 module，复杂度是消失，还是会重新扩散到多个调用方。

结论先行：**当前不建议做大规模重构**。项目第一阶段已经把最关键 seam 建起来了：`createApp/bootstrapApp/startServer`、`AuthProvider/AuthContext`、RBAC 模型、数据库初始化与 seed。继续大拆会增加风险。更合理的做法是保留现有结构，只做小步、可验证、能沉淀复用收益的重构。

## 2. 当前值得保留的设计

### 2.1 应用入口拆分

涉及文件：

- `src/app.js`
- `src/main.js`
- `src/server.js`

评估：应保留。

原因：

- `createApp()` 让测试可直接请求 Hono app。
- `bootstrapApp()` 集中处理配置和数据库初始化。
- `startServer()` 隔离监听端口、PID、进程信号等副作用。

删除测试：如果删除该拆分，测试、导入和启动副作用会重新混在一起，复杂度会扩散。

### 2.2 AuthProvider/AuthContext seam

涉及文件：

- `src/modules/auth/index.js`
- `src/modules/auth/auth-context.js`
- `src/modules/auth/providers/local-jwt.js`
- `src/middleware/auth.js`

评估：应保留，但暂不继续扩展 Logto 实现。

原因：

- 当前已有 Local JWT Adapter。
- 未来 Logto 可以作为第二个 Adapter 接入。
- 下游 RBAC、审计、菜单只依赖统一认证上下文。

注意：现在只有一个真实 Adapter，因此不应继续抽象出复杂 Provider Registry、动态插件系统或多租户身份映射层。

### 2.3 数据库 DDL 与 seed 分离

涉及文件：

- `src/db/index.js`
- `src/db/mysql-schema.js`
- `src/db/sqlite-schema.sql`
- `src/db/seed.js`

评估：应保留。

原因：

- SQLite 和 MySQL 都有可验证初始化路径。
- 默认 seed 同源，减少多项目重复配置。
- MySQL 已通过真实连接级测试。

不建议当前引入正式迁移框架，除非后续出现多版本升级、生产迁移回滚、多人协作变更冲突等需求。

### 2.4 RBAC 轻量管理接口

涉及文件：

- `src/routes/rbac.js`
- `src/controllers/RbacController.js`
- `src/models/RoleModel.js`
- `src/models/PermissionModel.js`

评估：应保留。

原因：

- 用户分角色、角色配权限是后台管理项目高频重复能力。
- 当前只覆盖高频闭环，没有扩展成重型 IAM。
- 写操作已记录审计日志。

不建议当前继续扩展角色/权限完整 CRUD、审批流、可视化权限设计器。

## 3. 候选重构评估

### 3.1 候选一：拆分 `UserController`

涉及文件：

- `src/controllers/UserController.js`
- `src/routes/auth.js`
- `src/routes/user-admin.js`

现状：

`UserController.js` 曾同时包含用户列表、用户 CRUD、认证、refresh、logout、当前用户、权限菜单、批量操作等多个用例。当前已按小步重构原则新增 `AuthController`，只迁移本地认证闭环，路由层继续保持 `auth.js` 和 `user-admin.js` 的既有边界。

问题：

- 认证流程和用户管理流程的变化原因不同。
- `register/login/refresh/logout` 与 `getUsers/updateUser/batchDelete` 共享同一个控制器文件，阅读成本高。
- 后续如果接入 Logto 或调整本地认证，会频繁打开同一个大文件，容易影响无关用户管理逻辑。

建议：**已完成第一步；后续不要继续大拆，先观察复用收益**。

推荐演进方式：

1. 已新增 `AuthController`，只迁移 `register/login/refreshToken/logout`。
2. 已保留现有 URL 契约 `/api/users/auth/*` 不变。
3. 已验证 `test-auth-session-flow.js`、`test-api.js`、`test:all`。
4. 只在认证控制器稳定并出现新的变化原因冲突后，再评估是否拆 `CurrentUserController` 或 `UserAdminController`。

不建议一次性拆成很多文件：

- 会增加 import 和命名负担。
- 容易出现为了“整洁”而损失 locality。
- 当前测试主要覆盖行为，不需要大规模移动代码来证明架构能力。

推荐强度：**Strong，但应小步执行**。

### 3.2 候选二：抽取审计记录 helper

涉及文件：

- `src/controllers/UserController.js`
- `src/controllers/RbacController.js`
- `src/models/AuditLogModel.js`

现状：

`UserController` 与 `RbacController` 都有从请求读取 IP/User-Agent、调用 `auditLogModel.record()` 的逻辑。两者目前重复不多，但语义一致。

问题：

- 如果后续新增业务资源，每个控制器都可能复制 `getRequestMeta` 和 `recordAudit`。
- 审计字段命名、失败策略、操作者 ID 规则可能分散。

建议：**值得探索，但等第三个写操作控制器出现时再抽**。

原因：

- 当前只有两个控制器使用，过早抽象可能只是移动代码。
- 一旦新增第二个业务资源模块，审计 helper 的复用收益会明显增加。

推荐目标形态：

```text
src/modules/audit/request-audit.js
  recordRequestAudit(c, entry)
```

它应隐藏：

- `x-forwarded-for` / `x-real-ip` 读取。
- `user-agent` 读取。
- `c.get('user')` / `authContext.localUserId` 兼容。
- 审计失败不阻断主流程的策略。

推荐强度：**Worth exploring，暂缓实现**。

### 3.3 候选三：抽取分页查询参数 helper

涉及文件：

- `src/controllers/UserController.js`
- `src/controllers/RbacController.js`
- `src/routes/*-schemas.js`

现状：

用户列表、角色列表、权限列表都有 `page/limit/search/status` 类参数。RBAC 控制器已改为读取 `validatedQuery`，用户控制器仍直接读取 `c.req.query()`。

问题：

- 查询参数默认值、最大 limit、search 规则分散。
- 后续新增后台资源列表时容易重复。

建议：**适合小步优化，但不急于做通用 Query Builder**。

推荐目标形态：

```text
src/modules/http/pagination.js
  getPaginationQuery(c, defaults)
```

或更轻量地先让 `UserController.getUsers()` 读取 `validatedQuery`。

不建议：

- 不要抽复杂筛选 DSL。
- 不要引入通用 Repository 查询构造器。
- 不要为少数列表接口做过度泛化。

推荐强度：**Worth exploring**。

### 3.4 候选四：抽取 JSON Schema 片段工厂

涉及文件：

- `src/routes/user-schemas.js`
- `src/routes/rbac-schemas.js`
- `src/middleware/validation/index.js`

现状：

多个 schema 重复定义整数 ID、分页、limit、search 等字段。

问题：

- 重复不算严重，但后续资源增多后会扩散。

建议：**暂不重构**。

原因：

- 当前 schema 显式展开，可读性好。
- JSON Schema 工厂容易让简单规则变得绕。
- 现在只有用户和 RBAC 两组 schema，不足以证明抽象收益。

可以先在新增第三个资源时再评估：

```text
src/routes/common-schemas.js
  idField
  paginationFields
  searchField
```

推荐强度：**Speculative，暂不做**。

### 3.5 候选五：引入正式迁移系统

涉及文件：

- `src/db/mysql-schema.js`
- `src/db/sqlite-schema.sql`
- `src/db/seed.js`
- `src/models/schema/*`

现状：

当前采用轻量幂等 DDL + seed，同步支持 SQLite/MySQL。

问题：

- 生产多版本升级时，单纯 `CREATE TABLE IF NOT EXISTS` 不足以表达字段变更、数据迁移、回滚策略。

建议：**当前不做，生产化前再做 ADR**。

原因：

- 项目目标是后台脚手架，不是数据库迁移平台。
- 当前已能支持新项目初始化和测试库验证。
- 引入迁移系统会增加学习和维护成本。

触发条件：

- 出现生产库版本升级需求。
- 出现字段 rename/drop/backfill。
- 多人并行维护 schema 变更。

推荐强度：**Speculative，暂不做**。

### 3.6 候选六：统一错误码与响应 builder

涉及文件：

- `src/middleware/error/index.js`
- `src/controllers/*`
- `src/utils/AppError.js`

现状：

项目已有全局错误处理中间件和 `AppError`。部分控制器直接构造成功响应，错误响应主要交给中间件。

建议：**暂不抽成功响应 builder**。

原因：

- 成功响应目前足够直观。
- 抽 `successResponse()` 这类 helper 容易变成浅 module，删除后复杂度并不会显著扩散。
- 真正值得沉淀的是接口契约文档和测试，而不是包装一层函数。

推荐强度：**Speculative，暂不做**。

## 4. 当前最推荐的重构路线

### 4.1 短期只做小步优化

优先级顺序：

1. **AuthController 小步拆分**：已完成，只迁移认证相关方法，不改 URL，不改响应结构。
2. **UserController 查询参数改用 validatedQuery**：先减少请求解析分散，不抽复杂 helper。
3. **等新增业务资源后再抽审计 helper**：当第三个控制器复制审计逻辑时再抽。

### 4.2 不建议当前做的事

- 不建议把所有控制器拆成很多小文件。
- 不建议抽通用 CRUD Controller。
- 不建议抽通用 Repository Builder。
- 不建议引入插件系统。
- 不建议为了“一致性”把所有响应都包一层 helper。
- 不建议现在引入正式数据库迁移框架。

## 5. 如果执行 AuthController 拆分，验收标准

必须满足：

- `/api/users/auth/register` 路径不变。
- `/api/users/auth/login` 路径不变。
- `/api/users/auth/refresh` 路径不变。
- `/api/users/auth/logout` 路径不变。
- refresh token 服务端会话行为不变。
- 登录失败日志和安全日志不丢。
- 注册、登录、刷新、登出测试全部通过。

建议测试：

```text
bun run tests/test-auth-session-flow.js
bun run tests/test-auth-provider.js
bun run tests/test-api.js
bun run tests/run-tests.js
```

## 6. 重构决策结论

当前项目已经具备较好的主线结构，不需要大规模重构。真正值得做的是“深而小”的重构：把变化原因明显不同、复用收益明显、测试能锁住行为的 module 拆出来。

本轮结论：

| 候选 | 是否建议 | 时机 |
| --- | --- | --- |
| 拆 `AuthController` | 已完成 | 已按小步方式执行 |
| 抽审计 helper | 暂缓 | 第三个写操作控制器出现时 |
| 抽分页 helper | 可探索 | 新增业务资源前后 |
| 抽 schema 工厂 | 不建议 | 至少第三个资源后再看 |
| 引入迁移系统 | 不建议 | 生产多版本升级前再 ADR |
| 响应 builder | 不建议 | 当前收益不足 |

一句话建议：**下一步如果要动代码，优先只拆认证控制器；除此之外，先继续沉淀业务资源接入，而不是为了架构整洁继续泛化。**
