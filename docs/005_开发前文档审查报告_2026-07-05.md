# 开发前文档审查报告

> 审查目标：站在项目复用、快速落地、后台脚手架的角度，审查 `docs/` 文档是否能支撑后续开发。  
> 审查依据：现有代码、项目架构、`CLAUDE.md` 协作规范、后台管理系统行业最佳实践。  
> 审查结论：当前可以进入“框架化整理与补文档”阶段，但不建议直接启动大规模功能开发。必须先完成本文列出的门禁项。
> 外部身份服务评估：Logto 集成建议见 `docs/006_Logto身份认证集成评估_2026-07-05.md`。
> 认证落地设计：Logto 集成流程与不集成时的轻量认证/RBAC 增强方案见 `docs/007_Logto集成设计与轻量认证方案_2026-07-05.md`。

## 1. 总体结论

`docs/` 曾经形成两类不同方向的文档。为避免长期维护和误用，过时文档已经删除，当前只保留主线文档和开发前审查报告：

- `001_后台管理复用框架评估_2026-07-05.md`：符合当前主线，建议作为后续开发的最高优先级方案依据。
- 已删除的历史文档：`002_项目参考价值与小而美演进评估_2026-07-05.md`、`003_轻量化优化建议_2026-07-05.md`、`004_接口调试示例_2026-07-05.md`。它们包含 lite 工具箱、纯前端验证、极简 API、fake token 等早期思路，容易误导当前后台复用框架开发。

因此，开发前必须明确：

> 当前项目主线不是“砍功能做 lite”，而是“沉淀一个轻量、模块化、可裁剪、可跨项目复用的后台管理脚手架”。

用户、认证、RBAC、系统配置、健康检查、日志、安全中间件、数据库适配不是要删除的负担，而是后台管理项目重复开发成本最高、最值得框架化沉淀的能力。

## 2. 文档审查结果

## 3. 代码与文档一致性审查

### 3.1 当前代码真实结构

现有代码是典型单体后台脚手架结构：

```text
src/main.js                     # 应用启动、全局中间件、docs、metrics、路由挂载
src/routes/users.js             # 用户、认证、管理员、超级管理员路由
src/routes/health.js            # 健康检查路由
src/controllers/UserController.js
src/controllers/HealthController.js
src/models/UserModel.js
src/models/RoleModel.js
src/models/PermissionModel.js
src/models/SystemConfigModel.js
src/models/BaseModel.js
src/middleware/auth.js
src/middleware/error/index.js
src/middleware/security/index.js
src/middleware/validation/index.js
src/db/index.js
src/models/schema/sqlite.js
src/models/schema/mysql.js
```

这说明当前更接近“后台脚手架”，而不是“极简工具箱”。因此文档主线必须围绕后台复用，而不是围绕裁剪功能。

### 3.2 当前路由事实

当前主要路由包括：

```text
GET  /
GET  /health
GET  /docs
GET  /metrics
GET  /api/health
GET  /api/health/db
GET  /api/health/system
POST /api/users/auth/register
POST /api/users/auth/login
POST /api/users/auth/refresh
POST /api/users/auth/logout
GET  /api/users/me
GET  /api/users/stats
GET  /api/users
GET  /api/users/:id
PUT  /api/users/:id
POST /api/users
DELETE /api/users/:id
POST /api/users/:id/toggle-status
POST /api/users/:id/verify-email
```

另有两个批量接口在审查时曾返回 `NOT_IMPLEMENTED`：

```text
POST   /api/users/batch/update-status
DELETE /api/users/batch
```

处理结果：已实现批量更新状态和批量软删除，不能再回退为半成品接口。

### 3.3 当前数据模型事实

SQLite schema 已包含：

```text
users
system_config
roles
permissions
user_roles
role_permissions
user_permissions
```

MySQL schema 目前只包含：

```text
users
system_config
```

这与“支持 SQLite/MySQL 双数据库”的文档表达不一致。若要面向多项目复用，MySQL 必须补齐 RBAC 表、索引、初始化和种子数据；否则必须明确 MySQL 是实验支持。

### 3.4 当前测试事实

已有测试覆盖方向较好：

- 用户创建、认证、更新、搜索、状态切换、软删除、角色权限。
- 角色和权限创建、搜索、分配、继承、优先级、有效期。
- 系统配置读取、写入、删除、批量设置、缓存。
- API 级测试脚本。

但测试仍存在落地风险：

- 测试脚本是自定义 runner，不是标准 `bun test` 风格，后续 CI 集成需确认。
- API 测试文件引用了角色、权限、系统配置接口，但当前代码路由中未看到独立 `/roles`、`/permissions`、`/system-config` 路由，需要核对是否测试与实现脱节。
- MySQL 路径没有同等测试。
- 未实现批量接口没有明确测试策略。

## 4. 启动开发前必须补齐的门禁项

2. **补第一阶段实施规格**：已新增 `008_后台脚手架第一阶段实施规格_2026-07-05.md`，明确第一阶段范围、接口、模块边界、验收标准。

5. **明确 MySQL 支持级别**：补齐 schema，或文档声明 MySQL 暂为实验支持。
6. **修正安全默认值**：默认 JWT secret、CORS credentials、CSP unsafe-inline 必须在开发规格中给出处理策略。

## 5. 建议的第一阶段范围

第一阶段不应该继续扩功能，而应把已有能力整理成可复用脚手架。

### 5.1 第一阶段应做

- 明确模块结构：core/auth/users/rbac/system-config。
- 稳定接口契约：响应、分页、错误、认证、权限。
- 修复明显风险：未实现接口、安全默认值、MySQL 支持级别说明。
- 建立开发规范：新增资源流程、权限命名、测试要求、文档要求。

### 5.2 第一阶段不做

- 不做复杂多租户。
- 不做审批流。
- 不做队列任务中心。
- 不做完整前端管理后台。
- 不做插件市场。
- 不盲目拆 package 发布 npm。

## 6. 行业最佳实践对齐建议

### 6.1 脚手架复用

行业成熟脚手架通常具备：

- 清晰 quickstart。
- 统一错误响应。
- 统一分页响应。
- 认证与权限开箱可用。
- 数据库初始化可重复执行。
- 种子数据可重复执行。
- 新增资源有模板或指南。
- 测试命令稳定。

当前项目已有基础，但缺少正式接口契约、新增资源指南和前端契约。

### 6.2 权限系统

后台管理常见做法是：

- RBAC 作为默认权限模型。
- 权限点采用 `resource:action`。
- 超级管理员绕过普通权限检查，但必须审计。
- 角色权限用于默认授权，用户直接权限用于例外处理。
- deny/grant 优先级必须明确。

当前代码已有模型雏形，但文档和种子策略不足。

### 6.3 数据库访问

最佳实践是：

- ORM 优先。
- 禁止字段名拼接。
- 搜索、排序、过滤使用白名单。
- 分页在数据库执行。
- schema 与迁移可重复。

当前 `BaseModel`、`UserModel`、`RoleModel`、`PermissionModel` 中存在内存分页和部分 SQL 表达式风险，开发前必须纳入技术债清单。

## 7. 最终门禁结论

当前文档已经足以判断项目方向：**后台管理复用脚手架**。但还不足以直接启动大规模开发。

允许启动的工作：

- 文档收敛。
- 接口契约补齐。
- 第一阶段实施规格编写。
- 未实现接口处理。
- 安全默认值修正。
- SQLite 主路径稳定化。

暂不建议启动的工作：

- 大规模新功能开发。
- 新增复杂模块。
- 发布 npm package。
- 重构全部目录结构。
- 多租户、审批流、队列等高级能力。

一句话结论：

> 先把当前项目整理成“可解释、可测试、可接入、可裁剪”的后台脚手架，再进入功能扩展；否则文档方向、接口契约、数据库支持和权限规则不统一，会让后续每个项目继续重复踩坑。
