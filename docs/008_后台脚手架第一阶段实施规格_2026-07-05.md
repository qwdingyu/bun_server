# 后台脚手架第一阶段实施规格

> 目标：在不集成 Logto 的前提下，沿 `005_开发前文档审查报告` 和 `007_Logto集成设计与轻量认证方案` 推进本地轻量认证与 RBAC 的可复用框架化。  
> 原则：先整理 seam，再补安全闭环；先稳定本地认证，再为 Logto Adapter 留口；不做无序补丁，不做大而全重构。

## 1. 第一阶段范围

第一阶段只做“已有认证/RBAC 能力的框架化整理与安全闭环”，不新增复杂业务模块。

必须完成：

1. 抽象认证 Provider seam。
2. 输出统一 AuthContext。
3. 保持 Local JWT 为默认认证 Provider。
4. 保持现有路由行为兼容。
5. 补齐本地 refresh token/session 持久化设计与后续实现。
6. 明确 RBAC 判断顺序。
7. 处理未实现接口与安全默认值。
8. 补充接口契约和前端集成契约。

暂不做：

- Logto 接入实现。
- OIDC/OAuth/SAML 自研。
- 多租户。
- 审批流。
- 完整前端管理后台。
- npm package 发布。

第二切片在不引入 Logto 的前提下，复用现有 `user_sessions` 表补齐 refresh token 服务端状态闭环。

## 3. AuthProvider 契约

```text
AuthProvider
  name
  authenticateRequest(c) -> AuthResult
  optionalAuthenticateRequest(c) -> AuthResult | optional failure
```

认证成功：

```text
AuthResult
  success: true
  user
  tokenPayload
  authContext
```

认证失败：

```text
AuthResult
  success: false
  status
  error.code
  error.message
```

## 4. AuthContext 契约

```text
AuthContext
  provider
  subject
  localUserId
  email
  username
  displayName
  roles
  permissions
  organizationId
  rawClaims
```

说明：

- `provider` 当前为 `local`，未来可为 `logto`。
- `localUserId` 是本地业务用户 ID，RBAC 和审计日志优先使用它。
- `rawClaims` 保留原始 token claims，便于未来 Logto Adapter 使用。

## 5. refresh token/session 持久化规格

当前 `refreshTokenPair()` 只验证 refresh token 本身，没有持久化撤销能力。第一阶段统一复用现有 `user_sessions` 表来承载会话状态。

当前表：

```text
user_sessions
  id
  user_id
  token_hash
  refresh_token_hash
  device_info
  user_agent
  ip_address
  location
  is_active
  expires_at
  last_used_at
  created_at
```

模型契约：

```text
SessionModel
  createSession(...)
  findActiveByRefreshToken(refreshToken)
  rotateRefreshToken(sessionId, tokens)
  revokeByRefreshToken(refreshToken)
  revokeSession(sessionId)
```

会话规则：

1. 登录和注册成功后生成 access token 与 refresh token，并创建 `user_sessions` 记录。
2. access token 和 refresh token 都带 `jti`，避免同一秒同一用户生成重复 token。
3. refresh token 明文只返回客户端，服务端只保存 hash。
4. 刷新 token 时必须检查 session 是否存在、未过期、未撤销、用户仍 active。
5. 刷新成功后轮换 refresh token，旧 refresh token 立即失效。
6. 登出时按客户端提交的 refresh token 撤销对应 session。

## 6. RBAC 判断顺序

统一权限判断顺序：

```text
1. super_admin 直接通过。
2. user_permissions 显式 deny 命中则拒绝。
3. user_permissions 显式 grant 命中则通过。
4. role_permissions 命中则通过。
5. 默认拒绝。
```

权限命名：

```text
permission.name = resource:action
```

标准 action：

```text
list | read | create | update | delete | export | import | approve
```

## 7. 验收标准
