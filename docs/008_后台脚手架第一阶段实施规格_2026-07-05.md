## 3. AuthProvider 契约

```text
AuthProvider
  name
  authenticateRequest(c) -> AuthResult
  optionalAuthenticateRequest(c) -> AuthResult | optional failure
```

认证成功：

```text
AuthResult
  success: true
  user
  tokenPayload
  authContext
```

认证失败：

```text
AuthResult
  success: false
  status
  error.code
  error.message
```

## 4. AuthContext 契约

```text
AuthContext
  provider
  subject
  localUserId
  email
  username
  displayName
  roles
  permissions
  organizationId
  rawClaims
```

说明：

- `provider` 当前为 `local`，未来可为 `logto`。
- `localUserId` 是本地业务用户 ID，RBAC 和审计日志优先使用它。
- `rawClaims` 保留原始 token claims，便于未来 Logto Adapter 使用。

## 5. refresh token/session 持久化规格

当前 `refreshTokenPair()` 只验证 refresh token 本身，没有持久化撤销能力。第一阶段统一复用现有 `user_sessions` 表来承载会话状态。

当前表：

```text
user_sessions
  id
  user_id
  token_hash
  refresh_token_hash
  device_info
  user_agent
  ip_address
  location
  is_active
  expires_at
  last_used_at
  created_at
```

模型契约：

```text
SessionModel
  createSession(...)
  findActiveByRefreshToken(refreshToken)
  rotateRefreshToken(sessionId, tokens)
  revokeByRefreshToken(refreshToken)
  revokeSession(sessionId)
```

会话规则：

1. 登录和注册成功后生成 access token 与 refresh token，并创建 `user_sessions` 记录。
2. access token 和 refresh token 都带 `jti`，避免同一秒同一用户生成重复 token。
3. refresh token 明文只返回客户端，服务端只保存 hash。
4. 刷新 token 时必须检查 session 是否存在、未过期、未撤销、用户仍 active。
5. 刷新成功后轮换 refresh token，旧 refresh token 立即失效。
6. 登出时按客户端提交的 refresh token 撤销对应 session。

## 6. RBAC 判断顺序

统一权限判断顺序：

```text
1. super_admin 直接通过。
2. user_permissions 显式 deny 命中则拒绝。
3. user_permissions 显式 grant 命中则通过。
4. role_permissions 命中则通过。
5. 默认拒绝。
```

权限命名：

```text
permission.name = resource:action
```

标准 action：

```text
list | read | create | update | delete | export | import | approve
```
