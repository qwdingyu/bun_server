# Logto 身份认证集成评估

> 评估目标：判断 Logto 是否适合与当前后台管理复用框架集成，避免重复发明认证、OIDC、OAuth、SSO、MFA、组织、多租户等轮子。  
> 评估对象：https://github.com/logto-io/logto  
> 评估方式：已克隆 Logto 仓库到临时目录，阅读 README、Docker Compose、Dockerfile、package、core/api/schemas 结构与 Management API SDK。  
> 评估结论：Logto 值得集成，但不适合嵌入到当前框架内部；推荐作为外部身份服务，通过 OIDC/JWKS/Management API 与当前框架对接。

## 1. 总体结论

Logto 是成熟的开源身份基础设施，不是一个可以轻量嵌入到 Bun/Hono 项目里的小型认证库。

它适合解决：

- 登录、注册、找回密码、MFA。
- OIDC、OAuth 2.1、SAML。
- 社交登录、企业 SSO。
- 多租户与组织。
- 机器到机器访问。
- Management API。
- 应用、资源、scope、角色等身份域管理。

它不适合直接替代：

- 当前框架的后台业务 CRUD。
- 当前框架的本地 SQLite 快速启动能力。
- 当前框架面向中小后台的资源管理脚手架。
- 当前框架内部的业务权限、菜单权限、按钮权限、数据权限全部语义。

专业建议：

> 若你的后台项目未来需要统一登录、企业 SSO、多应用账号体系、多租户、MFA，应该集成 Logto，避免自研认证体系；但不要把 Logto 源码嵌入本项目，也不要让当前脚手架强依赖 Logto。正确做法是把 Logto 做成可选 Auth Provider Adapter。

## 2. Logto 项目事实

### 2.1 项目定位

Logto README 明确定位为：

> modern, open-source auth infrastructure for SaaS and AI apps

核心能力包括：

- OIDC。
- OAuth 2.1。
- SAML。
- Multi-tenancy。
- Enterprise SSO。
- RBAC。
- 30+ SDK。
- Management API。

这说明它是身份基础设施，而不是后台 CRUD 框架。

### 2.2 技术栈与运行形态

从仓库实际结构看：

- monorepo，包含 `packages/core`、`packages/api`、`packages/console`、`packages/schemas`、`packages/cli` 等。
- Node.js 版本要求为 `^22.14.0`。
- 包管理器要求 pnpm。
- 后端核心使用 Koa，不是 Hono。
- 数据库依赖 PostgreSQL。
- OIDC 基于 `oidc-provider`。
- Docker Compose 需要 Logto app + PostgreSQL。
- Console 是 React/Vite 管理界面。

这些事实说明：Logto 本身是一个完整系统，运行成本明显高于本项目当前 Bun + Hono + SQLite 的脚手架定位。

### 2.3 集成相关能力

Logto 提供 `@logto/api`，用于调用 Management API。

其 `packages/api/README.md` 展示的基本用法是：

```ts
import { createManagementApi } from '@logto/api/management';

const { apiClient } = createManagementApi('default', {
  clientId: 'your-client-id',
  clientSecret: 'your-client-secret',
  baseUrl: 'https://your-logto-instance.com',
  apiIndicator: 'https://your-logto-instance.com/api',
});

const response = await apiClient.GET('/api/users');
```

这对当前框架很有价值：我们可以把 Logto 当外部身份源，通过 Management API 同步用户、角色、组织信息，或把身份管理交给 Logto。

## 3. 是否可以和当前框架集成

可以，但应采用“外部身份服务集成”，不是“源码嵌入”。

### 3.1 推荐集成方式

推荐形态：

```text
前端管理后台
  ↓ OIDC 登录
Logto 身份服务
  ↓ access token / id token
当前 Bun + Hono 后台框架
  ↓ JWKS 验证 token
业务 RBAC / 菜单权限 / 资源 CRUD
```

当前框架只负责：

- 校验 Logto 签发的 JWT。
- 解析用户 ID、email、name、organization、roles/scopes 等 claims。
- 将外部身份映射成本地用户上下文。
- 执行业务权限、菜单权限、数据权限。
- 提供业务 CRUD、系统配置、审计日志等后台能力。

Logto 负责：

- 登录体验。
- 注册流程。
- 密码安全。
- MFA。
- 社交登录。
- 企业 SSO。
- OIDC/OAuth/SAML 协议。
- 账号生命周期。

### 3.2 不推荐的集成方式

不推荐：

- 复制 Logto 源码到当前项目。
- 改造 Logto core 让它跑在 Hono/Bun 内部。
- 用 Logto 替换当前后台所有 RBAC 语义。
- 为了接入 Logto 强制所有项目都运行 PostgreSQL。
- 在当前脚手架默认安装 Logto 全套依赖。

原因：

- Logto 是 Koa + Node + PostgreSQL + pnpm monorepo，和当前 Bun/Hono/SQLite 轻量脚手架不是同一层级。
- 源码嵌入会引入巨大维护成本。
- Logto 版本升级、安全修复、协议合规都应交给上游维护。
- 当前框架的价值在于后台业务快速落地，不应变成身份平台 fork。

## 4. 是否是轻量级的好主意

结论分两层：

### 4.1 作为默认内置能力：不是轻量主意

如果把 Logto 作为当前框架默认依赖，要求所有后台项目都部署 Logto + PostgreSQL，那么它不轻量。

负担包括：

- 新增一个独立服务。
- 新增 PostgreSQL。
- 新增 OIDC 配置。
- 新增前端 OIDC 登录流程。
- 新增 Management API 的 M2M 配置。
- 本地开发复杂度上升。

这会损害当前框架“快速落地、低成本脚手架”的优势。

### 4.2 作为可选 Auth Provider：是好主意

如果把 Logto 做成可选 Adapter，它是很好的主意。

适用场景：

- 有多个后台项目需要统一账号。
- 需要企业 SSO。
- 需要 MFA。
- 需要社交登录。
- 需要多应用统一登录。
- 需要合规的 OIDC/OAuth 能力。
- 未来可能接入移动端、门户、客户后台。

此时引入 Logto 能大幅减少自研认证成本，是符合行业最佳实践的。

## 5. 当前框架与 Logto 的职责边界

### 5.1 Logto 应负责

- 身份认证。
- 用户登录体验。
- 密码与 MFA。
- 外部身份源连接。
- OIDC/OAuth/SAML 协议。
- 应用管理。
- 组织与成员基础模型。
- Access token / ID token 签发。
- JWKS。

### 5.2 当前框架应负责

- 后台业务资源 CRUD。
- 系统配置。
- 菜单权限。
- 按钮权限。
- 业务角色映射。
- 数据权限。
- 审计日志。
- 业务模块脚手架。
- SQLite-first 快速启动。
- 可选 MySQL 业务数据库。

### 5.3 双方重叠部分

重叠点主要是：

- 用户表。
- 角色。
- 权限。
- 组织/多租户。

处理原则：

> Logto 的用户、角色、组织是身份域；当前框架的用户、角色、权限是业务后台域。两者可以映射，但不要混为一张表、一个模型。

建议保留本地用户镜像表：

```text
external_identity_provider: logto
external_user_id: Logto user id
username/email/display_name/avatar
status
last_login_at
created_at/updated_at
```

本地业务权限仍可使用：

```text
roles
permissions
user_roles
role_permissions
user_permissions
```

也可以在高级模式下将 Logto roles/scopes 映射成本地 permissions。

## 6. 建议的 Adapter 设计

### 6.1 Auth Provider Interface

当前框架应抽象一个 Auth Provider Interface：

```text
AuthProvider
  verifyRequest(request) -> AuthContext
  getUser(externalUserId) -> ExternalUser
  syncUser(authContext) -> LocalUser
```

默认 Adapter：

```text
LocalJwtAuthProvider
```

Logto Adapter：

```text
LogtoAuthProvider
```

这样当前框架可以保持轻量默认路径，同时允许项目按需开启 Logto。

### 6.2 Logto 配置项

建议配置：

```yaml
auth:
  provider: local # local | logto
  logto:
    issuer: https://your-logto-domain/oidc
    jwks_uri: https://your-logto-domain/oidc/jwks
    audience: https://your-api-resource
    tenant_id: default
    management_api:
      enabled: false
      base_url: https://your-logto-domain
      api_indicator: https://your-logto-domain/api
      client_id: ${LOGTO_M2M_CLIENT_ID}
      client_secret: ${LOGTO_M2M_CLIENT_SECRET}
```

敏感信息必须通过环境变量注入，禁止写入 `config.yaml`。

### 6.3 请求认证流程

```text
1. 前端从 Logto 登录并拿到 access token。
2. 前端请求当前后台 API，携带 Authorization: Bearer <token>。
3. 后台从 Logto JWKS 获取公钥并缓存。
4. 后台验证 token issuer、audience、exp、signature。
5. 后台提取 sub/email/name/roles/scopes/organization。
6. 后台查找或创建本地用户镜像。
7. 后台执行本地业务权限检查。
```

### 6.4 Management API 使用场景

Management API 不应在每个请求中调用，只用于管理和同步场景：

- 管理 Logto 用户。
- 同步用户资料。
- 创建/读取应用。
- 同步组织成员。
- 同步 Logto roles/scopes。

常规 API 鉴权只需要 JWKS 验证 JWT，不需要调用 Management API。

## 7. 与当前代码的落地影响

### 7.1 需要调整的模块

当前代码中认证能力分散在：

- `src/middleware/auth.js`
- `src/utils/jwt/index.js`
- `src/controllers/UserController.js`
- `src/models/UserModel.js`
- `src/routes/users.js`

接入 Logto 前，应先抽象认证 seam：

```text
src/modules/auth/providers/local-jwt.js
src/modules/auth/providers/logto.js
src/modules/auth/auth-context.js
src/modules/auth/middleware.js
```

不要直接在现有 `authMiddleware` 中硬塞 Logto 逻辑，否则会让本地 JWT 与 Logto 混杂，后续难维护。

### 7.2 数据库影响

如果仅做 token 验证，不需要新增数据库。

如果做用户镜像，建议扩展 `users` 表字段：

```text
identity_provider
external_user_id
display_name
avatar_url
last_synced_at
```

或新增独立表：

```text
user_identities
  id
  user_id
  provider
  external_user_id
  claims_json
  last_synced_at
```

推荐后者，因为未来可能支持多个身份源。

### 7.3 权限影响

本地 RBAC 不应立即删除。

建议保留本地权限系统，并增加映射层：

```text
Logto scope / role / organization role
  ↓ mapping
local permission / local role
```

这样业务项目仍然能使用统一的 `requirePermissions`、`requireResourcePermission`。

## 8. 风险评估

### 8.1 技术风险

- Bun 运行时与 Logto SDK 的 Node 生态兼容性需要验证，尤其是 `@logto/api`、`openapi-fetch`、OAuth token 请求。
- JWKS 验证建议优先使用成熟 `jose`，但当前项目没有 `jose` 依赖，需评估是否引入。
- Logto OSS 需要 PostgreSQL；如果团队没有 PostgreSQL 运维基础，会增加复杂度。

### 8.2 产品风险

- 对很小的单后台项目，Logto 可能过重。
- 如果只是 admin/admin 登录，Logto 不是第一阶段必需。
- 如果强制所有项目使用 Logto，会降低脚手架落地速度。

### 8.3 维护风险

- 不要 fork Logto。
- 不要复制 Logto 内部表结构。
- 不要依赖 Logto 内部源码路径。
- 只依赖公开协议、公开 SDK、公开 Management API。

## 9. 推荐路线图

### Phase 0：保持当前本地认证可用

目标：不破坏当前快速落地能力。

- 保留 Local JWT 作为默认 provider。
- 修正默认 JWT secret 与安全配置。
- 先整理 Auth Provider Interface。

### Phase 1：增加 Logto JWT 验证 Adapter

目标：让当前框架可以接受 Logto access token。

- 新增 `LogtoAuthProvider`。
- 使用 JWKS 验证 token。
- 校验 issuer、audience、exp。
- 生成统一 `AuthContext`。
- 不接 Management API。

### Phase 2：增加用户镜像与首次登录同步

目标：让业务后台有本地用户上下文。

- 新增 `user_identities` 表。
- 首次请求时创建本地用户镜像。
- 后台展示用户时使用本地镜像。

### Phase 3：增加 Logto Management API 管理能力

目标：需要时管理外部用户、组织、应用。

- 使用 `@logto/api` 或轻量 HTTP client 调用 Management API。
- M2M client id/secret 仅放环境变量。
- 只在管理操作和同步任务中调用。

### Phase 4：角色/权限映射

目标：统一身份域与业务域权限。

- 定义 Logto scope 到本地 permission 的映射。
- 定义 Logto organization role 到本地 role 的映射。
- 支持缓存与手动同步。

## 10. 最终建议

专业结论：

1. **Logto 值得集成**：它能避免自研 OIDC/OAuth/SSO/MFA/组织等复杂身份能力。
2. **不要嵌入源码**：它是完整身份服务，不是轻量库；源码嵌入会严重增加维护成本。
3. **不要默认强依赖**：默认仍应保留 Local JWT，保证脚手架快速落地。
4. **做成可选 Adapter**：`auth.provider = local | logto` 是最合适的架构。
5. **本地 RBAC 不应删除**：Logto 负责身份域，本项目负责业务后台域，两者通过映射集成。
6. **先做 JWT 验证，后做 Management API**：不要一开始就接入全套 Logto 管理能力。
7. **适合多项目统一账号场景**：如果你有多个后台项目，Logto 是长期省成本的好选择。

一句话判断：

> Logto 不是“轻量内置组件”，但作为“可选外部身份服务”是非常好的主意；它能让我们少造认证和身份协议的轮子，同时保留当前 Bun/Hono 后台脚手架的快速落地优势。
