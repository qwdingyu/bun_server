# RBAC权限模型与种子数据

> 目标：统一后台脚手架默认角色、默认权限、角色权限映射、系统配置和默认管理员 seed，避免测试脚本、运行时初始化、文档三处各写一套，降低多项目复用时的漂移风险。

## 1. 当前结论

默认 seed 的唯一代码来源：

```text
src/db/seed.js
```

复用方：

```text
src/db/index.js       # 运行时 SQLite/MySQL 初始化后默认写入 seed
tests/init-db.js      # 测试数据库重建后复用同一套 SQLite seed
tests/test-mysql-init-plan.js  # 无 MySQL 服务时验证 MySQL seed 复用一致性
tests/test-mysql-integration.js # 有 MySQL 测试库时执行真实连接级 seed 验证
```

原则：

- SQLite 第一阶段完整支持默认 seed。
- MySQL 复用同源默认 seed，并通过 `seedMysqlDefaults()` 保持角色、权限、角色权限、管理员和系统配置 parity。
- seed 必须幂等，多次执行不会重复插入角色、权限、角色权限或管理员角色绑定。
- 默认管理员仅用于本地开发和测试；生产环境必须修改密码或禁用默认账号。

## 2. 默认角色

| 名称 | 显示名 | 级别 | 用途 |
| --- | --- | --- | --- |
| `super_admin` | 超级管理员 | 100 | 系统最高权限角色，默认拥有全部权限 |
| `admin` | 管理员 | 50 | 常规后台管理员，拥有用户、角色、权限、配置的只读和部分管理能力 |
| `user` | 普通用户 | 10 | 标准用户基础能力 |
| `guest` | 访客 | 1 | 预留最低权限角色，默认不分配受保护权限 |

## 3. 默认权限命名

权限命名统一使用：

```text
resource:action
```

当前默认权限：

```text
users:create
users:read
users:update
users:delete
users:list
users:manage
roles:create
roles:read
roles:update
roles:delete
roles:list
roles:assign
permissions:create
permissions:read
permissions:update
permissions:delete
permissions:list
permissions:assign
config:read
config:update
```

说明：

- `users:manage` 用于批量状态、批量管理等聚合能力。
- `roles:assign` 和 `permissions:assign` 用于分配类操作。
- 后续菜单、按钮、审计日志扩展时继续沿用 `resource:action`。

## 4. 默认角色权限映射

```text
super_admin -> 所有默认权限
admin       -> users:read, users:update, users:list, users:manage,
               roles:read, roles:list,
               permissions:read, permissions:list,
               config:read
user        -> users:read
guest       -> 空
```

注意：

- `super_admin` 当前通过角色映射拥有全部默认权限，同时路由中仍可通过 `requireRoles('super_admin')` 限制高风险批量操作。
- `admin` 默认不拥有 `users:delete`，避免普通管理员误删用户。
- `guest` 是预留角色，不作为公开匿名权限体系。

## 5. 默认管理员

默认管理员：

```text
username: admin
email: admin@example.com
password: admin
role: super_admin
```

安全要求：

- 仅允许本地开发和测试快速启动。
- 生产环境上线前必须修改密码、禁用默认账号或通过项目专属 seed 覆盖。
- 后续若增加生产安装脚本，应要求交互式创建管理员，而不是静默使用默认密码。

## 7. 幂等策略

SQLite 与 MySQL 使用同源 seed，不同方言使用各自幂等语法：

- `seedSqliteDefaults()` 使用 SQLite `ON CONFLICT ... DO UPDATE`。
- `seedMysqlDefaults()` 使用 MySQL `ON DUPLICATE KEY UPDATE`。
- `roles.name` 冲突时更新内置角色信息。
- `permissions.name` 冲突时更新内置权限信息。
- `role_permissions(role_id, permission_id)` 冲突时重新激活。
- `users.username` 冲突时更新默认管理员基础信息，不覆盖业务用户身份。
- `user_roles(user_id, role_id)` 冲突时重新激活。
- `system_config.config_key` 冲突时更新默认值。

## 8. 运行时策略

运行时 SQLite 初始化流程：

```text
1. 打开 SQLite。
2. 执行 sqlite-schema.sql。
3. 默认执行 seedSqliteDefaults()。
4. 如需跳过默认 seed，可设置 DB_SEED_DEFAULTS=false。
```

运行时 MySQL 初始化流程：

```text
1. 创建 mysql2 pool。
2. 默认执行 applyMysqlSchema(pool)。
3. 默认执行 seedMysqlDefaults(pool)。
4. 创建 Drizzle MySQL 实例。
```

MySQL 可通过 `DB_AUTO_MIGRATE=false` 跳过自动 DDL。

跳过 seed 适用于：

- 迁移已有生产数据库。
- 项目自己维护完整 seed。
- 测试特定空库场景。

## 9. 测试策略

当前 SQLite 测试初始化：

```text
bun run tests/init-db.js
```

该脚本只做三件事：

1. 删除测试 SQLite 文件。
2. 执行 `src/db/sqlite-schema.sql`。
3. 调用 `src/db/seed.js` 写入默认 seed。

MySQL 初始化测试：

```text
bun run tests/test-mysql-schema.js
bun run tests/test-mysql-init-plan.js
MYSQL_TEST_DATABASE_URL="mysql://user:pass@127.0.0.1:3306/bun_server_test" bun run test:mysql
```

其中真实连接级测试需要外部 MySQL 测试库，默认全量测试不强制执行。

不再允许：

- 在测试初始化脚本里复制整套 DDL。
- 在多个测试文件各自维护默认角色/权限列表。
- 写一次性临时脚本，调完删除，导致知识不沉淀。

## 10. 后续扩展建议

短期：

- 增加菜单/按钮权限 seed，例如 `menus:list`、`buttons:use` 或直接沿用业务资源权限。
- 为审计日志补标准 action，例如 `audit:list`、`audit:read`。

中期：

- 将 `bun run test:mysql` 接入 CI 的临时 MySQL 测试库，形成真实连接级发布门禁。
- 提供项目级 seed overlay，例如 `src/db/project-seed.js`，只追加业务权限，不修改核心 seed。
- 如后续需要版本化升级/回滚，再引入 Drizzle Kit 或专用迁移工具。

不建议：

- 为权限引入复杂策略语言。
- 在第一阶段做可视化权限编排器。
- 让前端自行定义权限名再倒灌后端。
