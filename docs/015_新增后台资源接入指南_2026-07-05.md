# 新增后台资源接入指南

> 目标：让多个后台项目新增业务资源时，有统一、轻量、可复制的落地流程，避免每个项目重新设计 CRUD、权限、菜单、审计、测试和文档。

## 1. 适用范围

本文适用于典型后台资源，例如：订单、产品、客户、设备、文章、字典、审批单等。

资源接入必须遵循本项目定位：

- 不做大而全生成器。
- 不引入重型领域框架。
- 不把业务逻辑塞进核心内核。
- 复用现有 Auth、RBAC、BaseModel、菜单、审计、错误响应和测试约定。

## 2. 推荐目录结构

以 `products` 资源为例：

```text
src/models/ProductModel.js              # 数据访问与领域校验
src/controllers/ProductController.js     # HTTP 编排、审计、响应格式
src/routes/products.js                   # 路由、认证、权限、参数校验
src/models/schema/sqlite.js              # SQLite Drizzle 表映射
src/models/schema/mysql.js               # MySQL Drizzle 表映射
src/db/sqlite-schema.sql                 # SQLite DDL
src/db/mysql-schema.js                   # MySQL DDL 与索引
tests/test-products.js                   # 资源接口/模型测试
docs/010_后台接口契约_2026-07-05.md      # 接口契约同步
docs/013_菜单按钮权限与审计日志_2026-07-05.md # 菜单/审计同步
```

如果资源只属于某个具体业务项目，不应污染核心模块；可以放在项目侧 `modules/products`，但仍遵循本文约定。

## 3. 接入步骤

### 3.1 定义资源边界

新增前必须先写清楚：

```text
资源名：products
业务含义：产品档案
是否核心通用：否，属于业务扩展
是否需要软删除：是
是否需要审计：新增、更新、删除、批量上下架需要
是否需要菜单：需要
是否需要导入导出：后续增强，不阻塞第一版
```

判断规则：

- 高频后台基础能力才进入核心内核。
- 具体业务资源默认作为业务扩展。
- 第一版只做列表、详情、新增、更新、删除、状态切换等闭环能力。

### 3.2 定义权限点

权限命名统一使用：

```text
resource:action
```

推荐最小权限集：

```text
products:list
products:read
products:create
products:update
products:delete
```

批量、导入、导出、审批等能力按需增加：

```text
products:batch_update
products:import
products:export
products:approve
```

要求：

- 权限点必须写入 seed 或项目级 seed overlay。
- 后端接口必须校验权限，前端菜单/按钮只用于体验，不是安全边界。
- 超级管理员可以通过角色放行高风险接口，但高风险操作必须审计。

### 3.3 定义数据库结构

每个资源至少同步四处：

1. SQLite DDL：`src/db/sqlite-schema.sql`
2. MySQL DDL：`src/db/mysql-schema.js`
3. SQLite Drizzle schema：`src/models/schema/sqlite.js`
4. MySQL Drizzle schema：`src/models/schema/mysql.js`

建议字段：

```text
id
name / title / code
status
created_at
updated_at
deleted_at（需要软删除时）
```

要求：

- 列表筛选字段必须有索引或明确数据量边界。
- SQLite 与 MySQL 字段语义保持一致。
- DDL 必须幂等，可重复初始化。
- 不允许只改 Drizzle schema，不补 DDL。

### 3.4 实现 Model

Model 负责：

- 字段校验。
- 唯一性校验。
- 查询过滤。
- 数据库分页。
- 软删除。
- 状态切换。

列表必须使用 `BaseModel.findPage()` 或等价数据库分页：

```js
const total = await this.count(queryFilter)
const products = await this.findPage(queryFilter, {
  page,
  limit,
  orderBy: 'created_at',
  order: 'desc',
  fields: this.safeFields
})
```

禁止：

```js
const rows = await this.findMany(filter)
const pageRows = rows.slice(offset, offset + limit)
```

原因：这会在数据量增长后变成隐藏性能坑。

### 3.5 实现 Controller

Controller 负责：

- 读取并校验当前用户。
- 调用 Model。
- 记录业务日志。
- 写入审计日志。
- 返回统一响应格式。

高价值变更必须审计：

```text
products.create
products.update
products.delete
products.batch_update_status
products.approve
```

审计字段建议：

```text
userId: 当前操作者
resourceType: products
resourceId: 资源 ID 或批量 ID
oldValues: 修改前关键字段
newValues: 修改后关键字段
action: products.update
```

### 3.6 实现 Routes

路由建议：

```text
GET    /api/products
GET    /api/products/:id
POST   /api/products
PUT    /api/products/:id
DELETE /api/products/:id
POST   /api/products/:id/toggle-status
```

路由规则：

- 静态路由必须放在 `/:id` 前面。
- 列表接口必须限制 `limit` 最大值，建议不超过 100。
- 写操作必须认证并校验权限。
- 高风险批量接口必须限制批量数量，并禁止操作当前操作者相关关键资源。

权限示例：

```js
productsRoutes.get('/', authMiddleware, requirePermissions('products:list'), productController.getProducts)
productsRoutes.post('/', authMiddleware, requirePermissions('products:create'), productController.createProduct)
productsRoutes.put('/:id', authMiddleware, requirePermissions('products:update'), productController.updateProduct)
productsRoutes.delete('/:id', authMiddleware, requirePermissions('products:delete'), productController.deleteProduct)
```

## 4. 菜单与按钮接入

在 `src/modules/rbac/menu.js` 增加菜单：

```js
{
  key: 'products',
  title: '产品管理',
  path: '/products',
  icon: 'box',
  permission: 'products:list',
  buttons: [
    { key: 'create', title: '新增产品', permission: 'products:create' },
    { key: 'update', title: '编辑产品', permission: 'products:update' },
    { key: 'delete', title: '删除产品', permission: 'products:delete' }
  ]
}
```

要求：

- 菜单权限必须和 seed 权限名一致。
- 页面级权限优先使用 `resource:list` 或 `resource:read`。
- 按钮权限使用具体 action。
- 后端仍必须做权限校验。

## 5. 测试清单

每个资源至少覆盖：

- 创建成功。
- 重复字段冲突。
- 列表分页。
- 搜索和筛选。
- 详情不存在返回 404。
- 更新成功。
- 删除或软删除成功。
- 权限不足返回 403。
- 菜单/按钮权限可见性。
- 高价值变更写入审计日志。

推荐测试命名：

```text
tests/test-products.js
```

测试数据规则：

- 使用唯一前缀或时间戳，避免共享测试库冲突。
- 测试完成后清理业务测试数据。
- 若需要数据库初始化，先跑 `bun run tests/init-db.js`。

## 6. 文档同步清单

新增资源时至少同步：

- `docs/010_后台接口契约_2026-07-05.md`：接口、请求、响应、权限。
- `docs/013_菜单按钮权限与审计日志_2026-07-05.md`：菜单和审计动作。
- 资源专项设计文档：复杂资源才需要，命名遵循 `序号_中文文件名_日期.md`。
- 踩坑记录：发现路由顺序、权限漂移、数据库差异、测试数据污染时必须记录。

## 7. 验收门禁

新增资源合并前必须确认：

```text
1. SQLite DDL、MySQL DDL、Drizzle schema 已同步。
2. seed 或项目级 seed overlay 已包含权限点。
3. 列表分页下推数据库。
4. 搜索、排序、过滤字段使用白名单或受控条件。
5. 写接口有认证和权限校验。
6. 高价值变更有审计日志。
7. 菜单/按钮权限已同步。
8. 接口契约已同步。
9. 相关测试通过。
10. 中文 commit 已提交。
```

## 8. 不建议第一版做的事

- 可视化表单设计器。
- 通用代码生成平台。
- 复杂 ABAC/策略语言。
- 数据权限表达式引擎。
- 数据库菜单表和动态菜单配置后台。
- 为单个业务资源引入新依赖。

这些能力会提高维护成本，不适合作为轻量复用脚手架的默认路径。

## 9. 踩坑记录

### 9.1 列表分页不能先查全量再切片

背景：用户、角色、权限列表早期普通查询分支曾先 `findMany()` 再 `slice()`。

原因：小数据量测试可以通过，但真实后台项目数据增长后会造成内存和响应时间风险。

排查过程：对照 `005_开发前文档审查报告_2026-07-05.md` 的技术债清单，定位到 `UserModel`、`RoleModel`、`PermissionModel` 的普通查询分支。

解决方案：新增 `BaseModel.findPage()` 并将三类列表改为数据库 `limit/offset`。

建议：新增资源列表必须默认使用数据库分页，代码评审中看到 `findMany()` 后 `slice()` 应直接退回。
