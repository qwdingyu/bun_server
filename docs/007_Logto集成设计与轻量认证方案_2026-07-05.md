# Logto 集成设计与轻量认证方案

> 目标：说明 Logto 的设计思路、运行流程、如何与当前后台管理复用框架有效集成；同时给出不集成 Logto 时，本项目如何以轻量方式增强登录、认证、RBAC。  
> 适用场景：多个后台项目复用、快速落地、认证与权限体系标准化。  
> 结论：Logto 适合作为可选外部身份服务；本项目应保留轻量本地认证作为默认路径，并通过 Auth Provider Adapter 同时支持两种模式。

## 1. 设计总原则

本项目的目标不是成为身份平台，而是成为后台管理复用脚手架。因此认证体系应遵循三条原则：

1. **默认轻量可跑**：不依赖外部身份服务时，项目仍可用本地账号、JWT、RBAC 快速启动。
2. **需要复杂身份能力时接入成熟产品**：OIDC、OAuth、SAML、MFA、企业 SSO、多应用统一登录不要自研，优先接入 Logto。
3. **身份域与业务域分离**：Logto 管身份，本项目管后台业务权限；两者通过用户镜像和权限映射衔接。

推荐最终形态：

```text
auth.provider = local  # 默认：轻量本地认证
auth.provider = logto  # 可选：外部 Logto 身份服务
```

## 2. Logto 的设计思路

### 2.1 Logto 是什么

Logto 是独立身份基础设施，核心价值是把复杂认证协议和账号安全能力产品化。

它解决的是身份域问题：

- 用户登录、注册、找回密码。
- OIDC / OAuth 2.1 / SAML。
- MFA。
- 社交登录。
- 企业 SSO。
- 应用管理。
- API Resource 与 scopes。
- 组织、多租户、组织角色。
- Management API。

它不是后台业务脚手架，也不负责你的业务 CRUD、菜单权限、按钮权限、数据权限。

### 2.2 Logto 的核心对象

理解 Logto 集成，需要先理解几个对象：

```text
User
  Logto 管理的身份用户。

Application
  接入 Logto 的前端、后端、机器应用。

API Resource
  一个受保护的后端 API 资源，通常对应本项目的后台 API。

Scope
  API Resource 下的权限范围，如 read:users、write:users。

Role
  Logto 身份域角色，可绑定 scopes。

Organization
  Logto 的组织/租户对象，可承载组织成员和组织角色。

Management API
  Logto 提供的管理接口，用于管理用户、应用、资源、角色等。
```

### 2.3 Logto 的运行形态

Logto 推荐作为独立服务运行：

```text
Logto App
  Node.js / Koa / oidc-provider

PostgreSQL
  Logto 元数据、用户、应用、角色、组织等

Logto Console
  管理界面
```

这也是为什么它不适合直接嵌入当前 Bun/Hono/SQLite 项目。

## 3. Logto 登录运行流程

典型后台管理登录流程如下：

```text
1. 管理后台前端发现用户未登录。
2. 前端跳转到 Logto 授权端点。
3. 用户在 Logto 完成登录、MFA、SSO 等流程。
4. Logto 回调前端配置的 redirect_uri。
5. 前端通过授权码换取 tokens。
6. 前端拿到 access token / id token。
7. 前端请求本项目 API，携带 Authorization: Bearer <access_token>。
8. 本项目后端通过 Logto JWKS 校验 token 签名和 claims。
9. 本项目生成统一 AuthContext。
10. 本项目执行本地业务 RBAC、菜单权限、按钮权限、数据权限。
```

关键点：

- 后端不要保存 Logto 用户密码。
- 后端不要自己处理 OIDC 授权码流程，交给前端 SDK 或 BFF。
- 后端只验证 access token，并根据 claims 建立业务上下文。
- 常规请求不应调用 Logto Management API，避免性能和稳定性问题。

## 4. 与当前框架的推荐集成架构

### 4.1 总体架构

```text
Admin Frontend
  ├─ 使用 Logto SDK 登录
  └─ 请求业务 API 时携带 access token

Logto Service
  ├─ 登录体验
  ├─ OIDC/OAuth/MFA/SSO
  ├─ JWKS
  └─ Management API

Bun + Hono Admin Framework
  ├─ LogtoAuthProvider 校验 JWT
  ├─ 用户镜像 user_identities
  ├─ 本地 RBAC / 菜单 / 按钮 / 数据权限
  ├─ 业务 CRUD
  └─ 审计日志
```

### 4.2 Auth Provider Adapter

建议抽象统一认证接口：

```text
AuthProvider
  name
  authenticate(request) -> AuthContext
  optionalAuthenticate(request) -> AuthContext | null
```

统一认证上下文：

```text
AuthContext
  provider: 'local' | 'logto'
  subject: string              # token sub 或本地用户 id
  localUserId: number | null
  email: string | null
  username: string | null
  displayName: string | null
  roles: string[]
  permissions: string[]
  organizationId: string | null
  rawClaims: object
```

默认实现：

```text
LocalJwtAuthProvider
  使用当前项目 JWT secret 验证本地 token。
```

Logto 实现：

```text
LogtoAuthProvider
  使用 Logto issuer / JWKS / audience 验证 access token。
```

### 4.3 建议目录结构

```text
src/modules/auth/
  index.js
  middleware.js
  auth-context.js
  providers/
    local-jwt.js
    logto.js
  services/
    user-sync.js
```

现有 `src/middleware/auth.js` 可以逐步收敛成对 `AuthProvider` 的调用，不要继续堆更多认证分支。

## 5. Logto 集成的落地步骤

### 5.1 Phase 0：准备 Logto

在 Logto Console 中配置：

1. 创建 SPA 或传统 Web 应用，用于后台前端登录。
2. 配置 redirect URI 和 post logout redirect URI。
3. 创建 API Resource，对应本项目后台 API。
4. 定义 scopes，例如：
   - `users:read`
   - `users:write`
   - `roles:read`
   - `roles:write`
5. 如需调用 Management API，创建 Machine-to-Machine 应用并授权。

### 5.2 Phase 1：后端只做 JWT 验证

这是最小可行集成。

配置项：

```yaml
auth:
  provider: logto
  logto:
    issuer: https://your-logto-domain/oidc
    audience: https://your-api-resource
    jwks_uri: https://your-logto-domain/oidc/jwks
```

后端处理：

```text
1. 从 Authorization header 提取 Bearer token。
2. 拉取并缓存 JWKS。
3. 使用公钥验证 JWT 签名。
4. 校验 iss、aud、exp、nbf。
5. 提取 sub、email、name、scope、organization_id 等 claims。
6. 生成 AuthContext。
7. 调用本地权限中间件。
```

推荐依赖：

```text
jose
```

理由：JWT/JWKS/OIDC token 校验是安全敏感能力，应使用成熟库，不要手写。

### 5.3 Phase 2：用户镜像

如果后台需要展示用户、绑定业务数据、做审计日志，建议建立本地用户镜像。

新增表：

```text
user_identities
  id
  user_id
  provider              # logto
  external_user_id      # Logto sub
  claims_json
  last_synced_at
  created_at
  updated_at
```

同步流程：

```text
1. LogtoAuthProvider 验证 token 成功。
2. 使用 sub 查询 user_identities。
3. 不存在则创建本地 users 记录和 identity 记录。
4. 存在则按策略更新 email、displayName、avatar。
5. 后续业务使用 localUserId。
```

### 5.4 Phase 3：权限映射

Logto 身份域权限不应直接替代本地业务权限，而应映射：

```text
Logto scope / role / organization role
  -> local role
  -> local permission
```

建议新增配置或表：

```text
external_permission_mappings
  id
  provider
  external_type         # scope | role | organization_role
  external_value
  local_role_id
  local_permission_id
  created_at
  updated_at
```

简单项目可以先用配置文件映射：

```yaml
auth:
  logto:
    permission_mapping:
      scopes:
        users:read: users:list
        users:write: users:update
      roles:
        admin: admin
        viewer: viewer
```

### 5.5 Phase 4：Management API

仅在需要后台管理 Logto 用户、应用、组织时引入。

使用场景：

- 管理 Logto 用户。
- 同步用户资料。
- 同步组织成员。
- 同步 Logto roles/scopes。
- 创建应用或 API Resource。

不建议：

- 每次业务请求都调用 Management API。
- 把 Management API 作为权限判断主路径。

## 6. 不集成 Logto 时的轻量认证增强方案

如果项目规模较小，或者只是单后台/内部后台，不集成 Logto 是合理的。此时应把当前本地认证体系增强到“够安全、够复用、够好维护”。

### 6.1 本地登录增强目标

本地模式应覆盖：

- 用户名/邮箱登录。
- 密码哈希。
- Access token + refresh token。
- 登出与 refresh token 失效。
- 用户禁用立即生效。
- 登录失败限流。
- 密码策略。
- RBAC 权限检查。
- 审计日志。

不建议第一阶段自研：

- OIDC Provider。
- SAML。
- 企业 SSO。
- 多因素认证全套流程。
- 社交登录连接器体系。

这些需求出现时应转向 Logto。

### 6.2 Refresh Token 设计

当前代码能生成 refresh token，但要成为可复用脚手架，建议增加持久化会话表：

```text
auth_sessions
  id
  user_id
  refresh_token_hash
  user_agent
  ip_address
  expires_at
  revoked_at
  created_at
  updated_at
```

流程：

```text
登录成功：
  1. 生成 access token，有效期短，例如 15 分钟。
  2. 生成 refresh token，有效期长，例如 7 天。
  3. 只保存 refresh token hash，不保存明文。

刷新 token：
  1. 校验 refresh token 签名或随机值。
  2. hash 后匹配 auth_sessions。
  3. 检查 expires_at、revoked_at、用户状态。
  4. 轮换 refresh token，旧 token 失效。

登出：
  1. 将当前 session revoked_at 置为当前时间。
```

### 6.3 密码与账号安全

建议：

- 使用 `Bun.password.hash` / `Bun.password.verify` 或成熟密码哈希库。
- 密码最小长度至少 8。
- 登录失败按用户和 IP 双维度限流。
- 管理员重置密码必须记录审计日志。
- 禁止在日志中输出密码、token、secret。

可选表：

```text
login_attempts
  id
  identifier
  ip_address
  success
  reason
  created_at
```

### 6.4 RBAC 轻量增强

当前已有 RBAC 表：

```text
roles
permissions
user_roles
role_permissions
user_permissions
```

建议补齐规则：

```text
permission.name = resource:action
resource = users | roles | permissions | system_config | ...
action = list | read | create | update | delete | export | import | approve
```

默认角色：

```text
super_admin
admin
operator
viewer
```

权限判断顺序建议：

```text
1. super_admin 直接通过。
2. 检查 user_permissions 中显式 deny，命中则拒绝。
3. 检查 user_permissions 中显式 grant，命中则通过。
4. 检查用户角色关联的 role_permissions，命中则通过。
5. 默认拒绝。
```

这样既支持常规角色授权，也支持个别用户例外。

### 6.5 权限缓存

为了避免每个请求多次查库，可以增加轻量缓存：

```text
permission_cache
  key: user:{id}:permissions
  ttl: 60s
```

失效时机：

- 用户角色变更。
- 角色权限变更。
- 用户直接权限变更。
- 用户禁用。

第一阶段可以先用内存 Map，后续如多实例部署再接 Redis。

### 6.6 菜单和按钮权限

后台管理落地时，接口权限不够，还需要前端菜单和按钮权限。

建议新增菜单表：

```text
menus
  id
  parent_id
  name
  title
  path
  icon
  sort
  permission
  visible
  status
```

运行流程：

```text
1. 前端登录后调用 GET /auth/me。
2. 前端调用 GET /auth/menus。
3. 后端根据用户 permissions 过滤菜单。
4. 前端根据 permissions 控制按钮显示。
```

### 6.7 审计日志

RBAC 与登录体系必须配审计日志，否则后台项目出问题难追踪。

建议表：

```text
audit_logs
  id
  actor_id
  action
  resource
  resource_id
  before_json
  after_json
  ip_address
  user_agent
  created_at
```

优先记录：

- 登录成功/失败。
- 登出。
- 创建用户。
- 禁用用户。
- 修改角色。
- 分配权限。
- 修改系统配置。

## 7. 两种方案的选择标准

### 7.1 选择 Logto 的情况

满足任一条件就建议接入 Logto：

- 多个后台项目需要统一账号。
- 需要企业 SSO。
- 需要社交登录。
- 需要 MFA。
- 需要 OIDC/OAuth 标准接入移动端、门户、第三方系统。
- 需要组织、多租户身份管理。
- 账号安全和合规要求较高。

### 7.2 选择本地轻量认证的情况

满足以下条件可以先不接 Logto：

- 单个内部后台。
- 用户量少。
- 不需要 SSO/MFA/社交登录。
- 部署环境希望 SQLite-first、零外部服务。
- 只需要 admin/operator/viewer 这类简单角色。
- 主要目标是快速上线业务 CRUD。

### 7.3 决策矩阵

| 维度 | 本地轻量认证 | Logto 集成 |
| --- | --- | --- |
| 启动成本 | 低 | 中高 |
| 外部依赖 | 无或很少 | Logto + PostgreSQL |
| OIDC/OAuth | 不建议自研 | 原生支持 |
| SSO/MFA | 不建议自研 | 原生支持 |
| 多项目统一账号 | 弱 | 强 |
| 单后台快速落地 | 强 | 中 |
| 长期身份治理 | 中 | 强 |
| 维护复杂协议 | 高风险 | 交给 Logto |

## 8. 推荐实施路线

### Phase 1：先强化本地认证 seam

目标：不管是否接 Logto，都先把认证边界理顺。

- 抽象 `AuthProvider`。
- 将现有 JWT 逻辑移动到 `LocalJwtAuthProvider`。
- 输出统一 `AuthContext`。
- `requireRoles`、`requirePermissions` 只依赖 `AuthContext` 和本地权限服务。

### Phase 2：补齐本地安全短板

- 持久化 refresh token。
- 登出撤销 session。
- 登录失败限流。
- RBAC 判断顺序文档化并测试。
- 默认角色和权限 seed。

### Phase 3：接入 Logto JWT 验证

- 新增 `LogtoAuthProvider`。
- 使用 JWKS 校验 access token。
- 支持用户镜像。
- 暂不接 Management API。

### Phase 4：按项目需要接 Management API

- 同步 Logto 用户。
- 同步组织与角色。
- 建立 scope/role 到本地权限映射。

## 9. 最终建议

1. 当前框架必须保留本地轻量认证，因为这是快速落地的默认能力。
2. Logto 不应成为默认强依赖，但应成为官方推荐的外部身份服务 Adapter。
3. 认证模块的第一优先级不是直接接 Logto，而是抽象 `AuthProvider` 和统一 `AuthContext`。
4. 不要自研 OIDC/OAuth/SAML/MFA/SSO；这些需求出现时直接使用 Logto。
5. 本地 RBAC 要继续强化，因为它负责业务后台权限，不应被 Logto 身份域完全替代。
6. 最优路线是：本地认证可用、Logto 可选、业务权限统一。

一句话结论：

> 轻量落地靠本地认证，长期省成本靠 Logto Adapter；两者不是二选一，而是通过 AuthProvider seam 共存。
