# 后台接口契约

> 目标：为多个后台管理项目复用提供稳定、轻量、可落地的接口约定。本文只记录当前框架主线契约，不包含 Logto 接入实现。

## 1. 通用响应格式

成功响应：

```json
{
  "success": true,
  "message": "操作成功",
  "data": {},
  "meta": {
    "timestamp": "2026-07-05T00:00:00.000Z",
    "duration": "12ms"
  }
}
```

错误响应：

```json
{
  "success": false,
  "error": {
    "code": "FORBIDDEN",
    "message": "权限不足",
    "timestamp": "2026-07-05T00:00:00.000Z",
    "requestId": "可选"
  }
}
```

分页响应放在业务数据内：

```json
{
  "success": true,
  "data": {
    "users": [],
    "pagination": {
      "page": 1,
      "limit": 20,
      "total": 0,
      "pages": 0
    },
    "search": null
  }
}
```

## 2. 认证约定

受保护接口使用 Bearer Token：

```text
Authorization: Bearer <accessToken>
```

本地认证默认返回：

```json
{
  "accessToken": "jwt",
  "refreshToken": "jwt",
  "refreshTokenExpiresAt": 1783180800,
  "expiresIn": "24h",
  "tokenType": "Bearer"
}
```

refresh token 规则：

- 明文只返回客户端。
- 服务端只保存 hash。
- 刷新时必须命中有效 `user_sessions`。
- 刷新成功会轮换 refresh token，旧 refresh token 立即失效。
- 登出时客户端应提交 refresh token 以撤销会话。

## 3. 认证接口

### 3.1 注册

```text
POST /api/users/auth/register
```

请求：

```json
{
  "username": "demo_user",
  "email": "demo@example.com",
  "password": "password123",
  "first_name": "Demo",
  "last_name": "User"
}
```

成功：返回用户信息与 token 对。

### 3.2 登录

```text
POST /api/users/auth/login
```

请求：

```json
{
  "identifier": "demo_user 或 demo@example.com",
  "password": "password123"
}
```

成功：返回用户信息与 token 对，并创建会话。

### 3.3 刷新令牌

```text
POST /api/users/auth/refresh
```

请求：

```json
{
  "refreshToken": "jwt"
}
```

成功：返回新的 token 对；旧 refresh token 不可再次使用。

### 3.4 登出

```text
POST /api/users/auth/logout
Authorization: Bearer <accessToken>
```

请求：

```json
{
  "refreshToken": "jwt"
}
```

说明：`refreshToken` 建议必传，用于撤销服务端会话；未传时只完成当前 access token 语义上的登出响应。

### 3.5 当前用户

```text
GET /api/users/me
Authorization: Bearer <accessToken>
```

成功：返回数据库中的最新用户安全字段。

### 3.6 当前用户有效权限

```text
GET /api/users/me/permissions
Authorization: Bearer <accessToken>
```

成功响应 `data`：

```json
{
  "permissions": [
    {
      "id": 1,
      "name": "users:read",
      "display_name": "查看用户",
      "resource": "users",
      "action": "read",
      "source": "role",
      "permission_type": "grant"
    }
  ],
  "permissionNames": ["users:read"]
}
```

说明：`permissionNames` 供前端路由、按钮和菜单快速判断；后端接口仍必须继续做权限校验。

### 3.7 当前用户菜单与按钮

```text
GET /api/users/me/menus
Authorization: Bearer <accessToken>
```

成功响应 `data`：

```json
{
  "menus": [
    {
      "key": "users",
      "title": "用户管理",
      "path": "/users",
      "icon": "users",
      "permission": "users:read",
      "buttons": [
        { "key": "create", "title": "新增用户", "permission": "users:create" }
      ]
    }
  ],
  "permissionNames": ["users:read", "users:create"]
}
```

说明：当前采用代码内静态菜单元数据，适合后台脚手架裁剪复用；暂不引入数据库菜单表和可视化菜单配置。

## 4. 用户接口

### 4.1 用户列表

```text
GET /api/users?page=1&limit=20&status=active&search=demo
```

说明：当前为公开接口；后续如用于生产后台，建议按项目启用认证和 `users:list` 权限。

### 4.2 用户详情

```text
GET /api/users/:id
Authorization: Bearer <accessToken>
```

### 4.3 更新用户

```text
PUT /api/users/:id
Authorization: Bearer <accessToken>
```

权限：本人或 `admin` / `super_admin`。

### 4.4 创建用户

```text
POST /api/users
Authorization: Bearer <accessToken>
```

权限：`admin` / `super_admin`。

### 4.5 删除用户

```text
DELETE /api/users/:id
Authorization: Bearer <accessToken>
```

权限：管理员；不能删除自己；执行软删除。

### 4.6 切换状态

```text
POST /api/users/:id/toggle-status
Authorization: Bearer <accessToken>
```

权限：管理员；不能修改自己。

### 4.7 验证邮箱

```text
POST /api/users/:id/verify-email
Authorization: Bearer <accessToken>
```

权限：本人或管理员。

### 4.8 批量更新状态

```text
POST /api/users/batch/update-status
Authorization: Bearer <accessToken>
```

权限：`super_admin`。

请求：

```json
{
  "userIds": [1, 2, 3],
  "status": "inactive"
}
```

约束：

- `userIds` 最多 100 个。
- 不能包含当前操作者。
- 只影响未软删除用户。
- 路由必须注册在 `/:id` 动态路由之前。

### 4.9 批量删除

```text
DELETE /api/users/batch
Authorization: Bearer <accessToken>
```

权限：`super_admin`。

请求：

```json
{
  "userIds": [1, 2, 3]
}
```

约束：

- `userIds` 最多 50 个。
- 不能包含当前操作者。
- 执行软删除，不做物理删除。
- 路由必须注册在 `/:id` 动态路由之前。

## 5. RBAC 权限契约

权限判断顺序：

```text
1. super_admin 角色可在中间件层直接放行角色类接口。
2. user_permissions 显式 deny 命中则拒绝。
3. user_permissions 显式 grant 命中则通过。
4. role_permissions 命中则通过。
5. 默认拒绝。
```

权限命名：

```text
resource:action
```

推荐 action：

```text
list | read | create | update | delete | export | import | approve
```

## 6. 安全默认值

- 生产环境必须配置安全 `JWT_SECRET` 或 `auth.jwt_secret`。
- 默认 CORS 不再使用 `origin: '*'` 与 `credentials: true` 组合。
- 默认 CSP 不再包含 `unsafe-inline`；如前端框架开发阶段确有需要，应由项目显式覆盖配置。
- 批量操作必须有管理员权限、审计日志和自操作保护。

## 7. MySQL 支持级别

当前第一阶段以 SQLite 为完整主线。MySQL 已补齐核心 Drizzle schema parity，但仍属于实验支持，原因：

- MySQL schema 已覆盖 `users/system_config/roles/permissions/user_roles/role_permissions/user_permissions/user_sessions/audit_logs`。
- 目前只做 schema 导出冒烟测试，尚未连接真实 MySQL 执行建表迁移。
- 默认 seed 仍只对 SQLite 落地，MySQL seed parity 尚未完成。
- 后续要宣称 MySQL 生产可用，必须补齐迁移脚本、seed parity 和集成测试矩阵。
