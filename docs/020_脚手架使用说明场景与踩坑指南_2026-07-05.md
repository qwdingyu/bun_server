# 脚手架使用说明场景与踩坑指南

日期：2026-07-05  
适用项目：`bun_server`  
定位：后台管理类项目复用脚手架，而不是大而全平台

## 1. 文档目的

本文面向后续使用本项目搭建新后台系统的开发者、产品负责人和项目负责人，回答四个问题：

1. 这个脚手架适合解决什么问题。
2. 复制到新项目后应该按什么顺序落地。
3. 哪些地方必须修改，哪些地方不要轻易改。
4. 当前项目已经踩过哪些坑，如何避免重复踩坑。

本文只基于当前仓库中已经存在并验证过的代码、脚本和文档，不假设尚未实现的能力。

## 2. 脚手架定位

### 2.1 核心目标

本项目的目标是成为一个“小而美的后台瑞士军刀”：

- 默认能启动：SQLite 开箱可用。
- 默认能测试：`bun run test:all` 可作为全量回归入口。
- 默认有后台高频能力：认证、用户、RBAC、菜单权限、审计日志、系统配置、健康检查。
- 默认可替换：数据库可选 SQLite/MySQL，认证保留 `AuthProvider` seam。
- 默认可裁剪：业务项目可以按需删减模块，但不要在脚手架内核里堆复杂平台能力。

### 2.2 不是哪些东西

当前项目不是：

- 完整 IAM 平台。
- 多租户/组织/岗位/数据权限一体化平台。
- 工作流或审批流引擎。
- 低代码后台生成器。
- 前后端一体的大型管理平台。
- 插件市场或模块市场。

这些能力如果某个业务项目确实需要，可以在具体项目中扩展，不建议默认压入脚手架内核。

## 3. 适用场景

### 3.1 非常适合

| 场景 | 原因 |
| --- | --- |
| 中小型后台管理系统 | 用户、角色、权限、审计、配置都是高频重复能力 |
| 内部运营管理后台 | 默认 RBAC、菜单权限、批量操作、审计日志可直接复用 |
| SaaS 管理端的基础 API | 可在现有用户/RBAC 基础上继续扩业务资源 |
| 需要快速验证业务闭环的项目 | SQLite 可本地快速启动，MySQL 可做真实部署验证 |
| 多个相似后台项目的统一底座 | 统一接口风格、测试入口、数据库初始化和权限模型 |

### 3.2 谨慎适用

| 场景 | 原因 | 建议 |
| --- | --- | --- |
| 强多租户系统 | 当前没有租户隔离模型 | 先写 ADR，再扩展 schema、权限和查询隔离 |
| 强审批流系统 | 当前没有流程引擎 | 不要硬塞进 RBAC，单独建业务模块 |
| 大规模 IAM 替代品 | 当前 RBAC 是后台轻量模型 | 评估 Logto/Auth0/Keycloak 等成熟方案 |
| 多数据库长期演进 | 当前未接入正式迁移框架 | 生产多版本升级前补迁移 ADR 和工具选型 |
| 高并发公网 API | 当前更偏后台管理 | 增加压测、限流策略、缓存和观测后再上线 |

### 3.3 不建议直接使用

- 需要复杂组织架构、岗位继承、跨租户授权的系统。
- 需要统一登录中心并服务很多外部应用的系统。
- 需要非常复杂审计合规、审批留痕、数据血缘的系统。
- 需要自动生成前端页面、表单、工作流的低代码平台。

## 4. 当前已经具备的可复用能力

### 4.1 应用入口

当前入口拆分为：

- `src/app.js`：`createApp()`，只负责创建 Hono app、挂载中间件和路由。
- `src/main.js`：`bootstrapApp()`，负责加载配置、校验配置、初始化数据库，并按选项启动服务。
- `src/server.js`：`startServer()`，负责监听端口、PID、进程信号处理。

复用价值：

- 测试可以直接调用 app，不必启动真实端口。
- 部署时可以单独关注 server 生命周期。
- 后续如果接入 serverless 或测试容器，改动范围更小。

### 4.2 认证与会话

当前默认认证是 Local JWT：

- `POST /api/users/auth/register`
- `POST /api/users/auth/login`
- `POST /api/users/auth/refresh`
- `POST /api/users/auth/logout`

当前已拆出：

- `src/controllers/AuthController.js`：注册、登录、refresh token 轮换、登出撤销。
- `src/modules/auth/index.js`：认证 Provider 入口，当前返回本地 JWT Provider。
- `src/modules/auth/providers/local-jwt.js`：本地 JWT Provider。

关键机制：

- 登录成功后签发 access token 和 refresh token。
- refresh token 会写入服务端会话表。
- refresh 成功后轮换 refresh token，旧 refresh token 失效。
- logout 如果传入 refresh token，会撤销对应服务端会话。

复用建议：

- 如果项目只需要后台本地账号体系，优先沿用 Local JWT。
- 如果需要外部身份提供商，优先通过 `AuthProvider` seam 扩展，不要把第三方 SDK 调用散落在控制器里。
- 不要改 `/api/users/auth/*` 路径，除非同步更新接口契约、前端契约和测试。

### 4.3 用户管理

当前用户管理覆盖：

- 当前用户信息。
- 当前用户权限。
- 当前用户菜单树。
- 用户列表分页。
- 用户详情。
- 创建用户。
- 更新用户。
- 软删除用户。
- 状态切换。
- 邮箱验证。
- 批量状态更新。
- 批量删除。

边界说明：

- `UserController` 继续承载用户管理、当前用户、权限菜单和批量操作。
- 当前不继续拆 `UserAdminController`，避免为了形式整洁增加维护成本。
- 后续只有当文件再次明显变大、变化原因冲突，并且测试能锁住行为时才继续拆。

### 4.4 RBAC 与菜单按钮权限

当前 RBAC 覆盖：

- 角色。
- 权限。
- 用户角色。
- 角色权限。
- 用户直授权限。
- deny 优先。
- 轻量 RBAC 管理接口。

默认角色包括：

- `super_admin`
- `admin`
- `manager`
- `user`

默认管理员见 `docs/012_RBAC权限模型与种子数据_2026-07-05.md`：

```text
username: admin
email: admin@example.com
password: admin
role: super_admin
```

注意事项：

- 默认管理员密码只能用于本地开发或初始化验证。
- 新项目上线前必须修改默认管理员密码或重建管理员账号。
- 高风险批量接口当前使用 `requireRoles('super_admin')` 限制。
- 不要把业务数据权限、审批权限、菜单权限混成一个巨大抽象；先满足后台常见 RBAC 即可。

### 4.5 数据库支持

当前支持：

- SQLite：默认配置，数据库文件为 `./data/app.db`。
- MySQL：通过 `DATABASE_TYPE=mysql` 和连接配置启用。

SQLite 初始化行为：

- 使用 `bun:sqlite`。
- 执行 `src/db/sqlite-schema.sql`。
- 默认写入 seed，除非 `DB_SEED_DEFAULTS=false`。
- 设置 `PRAGMA foreign_keys = ON`、`journal_mode = WAL`、`synchronous = NORMAL`。

MySQL 初始化行为：

- 使用 `mysql2/promise` pool。
- `DATABASE_URL` 优先级最高。
- 未设置 `DATABASE_URL` 时读取 `DB_HOST/DB_PORT/DB_USER/DB_PASSWORD/DB_NAME`。
- 默认执行 `applyMysqlSchema(pool)`，除非 `DB_AUTO_MIGRATE=false`。
- 默认执行 `seedMysqlDefaults(pool)`，除非 `DB_SEED_DEFAULTS=false`。

重要边界：

- 当前建表是幂等 DDL，不是完整迁移系统。
- 生产多版本升级前，应先评估正式迁移方案，不要只依赖自动建表。
- 测试 MySQL 必须使用测试库或临时库，不要指向生产库。

### 4.6 审计日志

当前审计覆盖用户和 RBAC 高风险写操作，例如：

- 用户更新、删除、状态切换、批量操作。
- RBAC 用户角色分配和移除。
- RBAC 角色权限授权和撤销。

复用建议：

- 新增业务资源时，涉及高风险写操作要补审计。
- 不要一开始抽通用审计 helper；等第三个控制器出现重复审计样板后再抽。
- 审计日志要记录操作者、动作、目标、结果和关键上下文，但不要记录明文密码、token 等敏感信息。

## 5. 新项目复用落地流程

### 5.1 第一步：复制脚手架

建议流程：

```text
1. 克隆或复制本项目到新项目目录。
2. 修改 package.json 的 name、description。
3. 修改 README 中的项目名称和业务说明。
4. 保留 docs 中的架构、契约、踩坑文档，再新增业务项目自己的文档。
```

不要一开始就删除大量模块。建议先跑通，再裁剪。

### 5.2 第二步：修改基础配置

必须检查：

| 项 | 文件/变量 | 说明 |
| --- | --- | --- |
| 端口 | `config.yaml` 的 `server.port` 或 `PORT` | 默认 `5050` |
| 数据库类型 | `database.type` 或 `DATABASE_TYPE` | 默认 `sqlite` |
| SQLite 路径 | `database.path` 或 `DATABASE_PATH` | 默认 `./data/app.db` |
| MySQL URL | `DATABASE_URL` | MySQL 连接 URL，优先级最高 |
| JWT 密钥 | `auth.jwt_secret` 或 `JWT_SECRET` | 生产环境必须改 |
| CORS | `cors.origin` 或 `CORS_ORIGIN` | 按前端域名配置 |
| 日志目录 | `logging.log_directory` 或 `LOG_DIRECTORY` | 默认 `./logs` |

生产环境注意：

- `NODE_ENV=production` 时不能使用默认 JWT 占位值。
- `JWT_SECRET` 必须使用足够随机的长密钥。
- 不要把真实数据库密码写进仓库。

### 5.3 第三步：初始化数据库

SQLite：

```bash
bun run db:init
```

MySQL：

```bash
DATABASE_TYPE=mysql \
DATABASE_URL="mysql://user:pass@127.0.0.1:3306/bun_server_test" \
bun run db:init
```

注意：

- MySQL 初始化会在目标库执行建表和 seed。
- 本地测试请使用专用库，例如 `bun_server_test`。
- 不要对生产库执行 `bun run db:reset`。
- `scripts/reset-db.mjs` 对 SQLite 会删除数据库文件；对 MySQL 只提示手工处理，不会自动 drop 远程库。

### 5.4 第四步：启动与健康检查

开发启动：

```bash
bun run dev
```

生产或普通启动：

```bash
bun run start
```

健康检查：

```bash
curl http://localhost:5050/
curl http://localhost:5050/api/health
```

如果端口已改，请同步替换 `5050`。

### 5.5 第五步：跑测试门禁

建议最低门禁：

```bash
bun run test:all
```

MySQL 专项测试：

```bash
MYSQL_TEST_DATABASE_URL="mysql://user:pass@127.0.0.1:3306/bun_server_test" bun run test:mysql
```

注意：

- `test:mysql` 必须指向测试库。
- 如果新增接口契约，至少补接口冒烟测试。
- 如果新增核心模型方法，至少补模型级测试。
- 测试脚本不能“内部失败但退出 0”。当前 `tests/test-config.js` 已修正这个问题，后续新增脚本也要遵守。

### 5.6 第六步：新增第一个业务资源

优先参考：

- `docs/015_新增后台资源接入指南_2026-07-05.md`
- `docs/019_最小业务资源模板_2026-07-05.md`

推荐顺序：

```text
1. 定义业务资源边界和字段。
2. 补 schema。
3. 补 model。
4. 补 controller。
5. 补 routes。
6. 补权限 seed。
7. 补菜单/按钮权限。
8. 补审计日志。
9. 补接口契约文档。
10. 补测试。
```

不要先写通用 CRUD Controller。当前项目明确不建议为了第一个业务资源抽大而全生成器。

## 6. 使用注意事项

### 6.1 必须修改的内容

新项目上线前必须处理：

- 修改 `JWT_SECRET`。
- 修改默认管理员账号或密码。
- 配置真实 CORS 域名。
- 确认数据库类型和连接信息。
- 确认日志目录和部署路径权限。
- 确认是否允许自动建表和默认 seed。
- 同步 README 和 docs 中的项目说明。

### 6.2 不要轻易修改的内容

除非有明确原因，不建议改：

- `/api/users/auth/*` 认证路径。
- 统一 JSON 响应基本结构。
- refresh token 服务端会话机制。
- RBAC 的 deny 优先规则。
- `createApp/bootstrapApp/startServer` 的入口分层。
- `test:all` 作为全量测试入口。

这些属于脚手架复用稳定契约，修改后会影响多个项目之间的迁移成本。

### 6.3 可以按项目裁剪的内容

如果某个项目不需要，可以在确认测试后裁剪：

- RBAC 管理接口。
- 系统配置模型。
- 批量用户操作。
- MySQL 支持或 SQLite 支持之一。
- 菜单树接口。

裁剪原则：

- 先写清楚为什么裁剪。
- 先跑测试确认裁剪前行为。
- 裁剪后同步 docs。
- 不要只删代码不删文档。

## 7. 已踩坑记录

### 7.1 SQLite DDL 不要手动按分号拆分

背景：

SQLite schema 文件中包含多条 DDL、行注释和块注释。

原因：

如果手动按分号拆分 SQL，再逐条执行，容易被注释干扰，导致建表语句被跳过、索引先执行，甚至残留 `*/` 被当成 SQL。

当前解决方案：

- `src/db/index.js` 中对 SQLite 使用 `sqlite.exec(ddl)` 一次执行完整 schema 文件。

建议：

- 不要重新引入手写 SQL 分割器。
- 如果未来 schema 更复杂，应引入正式迁移工具，而不是自己解析 SQL。

### 7.2 MySQL 测试必须限定测试库

背景：

MySQL 真实连接测试会执行建表和默认 seed。

风险：

如果连接到错误数据库，可能污染非测试库。当前脚本不应该用于生产库验证。

当前做法：

- 使用 `MYSQL_TEST_DATABASE_URL` 指向专用测试库。
- 文档中明确禁止指向生产库。
- 已有验证记录见 `docs/014_MySQL初始化与种子数据_2026-07-05.md`。

建议：

- 本机测试使用 `bun_server_test` 之类独立库。
- 不要在自动化脚本里写 `DROP DATABASE`。
- 生产库迁移必须单独评估迁移方案。

### 7.3 测试脚本不能误报通过

背景：

曾发现 `tests/test-config.js` 内部有失败，但脚本退出码仍为 0，导致全量测试入口可能误判为通过。

原因：

测试脚本只打印失败摘要，没有在 `testRunner.failed > 0` 时 `process.exit(1)`。

当前解决方案：

- `tests/test-config.js` 已在末尾增加失败退出。
- 配置缓存测试改为通过 `getDrizzleInstance()` 和 Drizzle schema 绕过模型缓存，验证 `refreshCache()` 行为。

建议：

- 新增测试脚本必须确保失败时返回非 0。
- 不要只看控制台文字“失败”，还要看命令退出码。
- 全量测试入口应持续保持可信。

### 7.4 路由顺序不能随意调整

背景：

用户路由中同时存在静态路径和动态路径，例如：

- `/me`
- `/stats`
- `/batch/update-status`
- `/:id`

风险：

如果把 `/:id` 放到静态路径前面，静态路径可能被误匹配为动态参数。

当前做法：

- `src/routes/user-admin.js` 中静态路由和批量路由位于 `/:id` 前。

建议：

- 新增类似 `/export`、`/import`、`/batch/*` 路由时，必须放在 `/:id` 之前。
- 路由调整后必须跑 `bun run tests/test-api.js` 或相关路由测试。

### 7.5 认证控制器拆分要保持 URL 契约不变

背景：

认证逻辑原来在 `UserController` 中，后续接入第三方认证或调整本地认证会影响用户管理代码。

当前解决方案：

- 已新增 `AuthController`。
- `src/routes/auth.js` 继续挂载在 `/api/users/auth/*`。
- `UserController` 继续负责用户管理，不继续大拆。

建议：

- 控制器拆分是内部结构优化，不应该影响外部 API。
- 拆分后必须跑认证会话流测试和 API 契约测试。
- 不要借重构机会改响应字段。

### 7.6 默认 seed 是便利，不是生产安全策略

背景：

默认 seed 会创建角色、权限、管理员和系统配置，方便脚手架开箱验证。

风险：

默认管理员账号密码如果进入生产，会造成安全风险。

当前事实：

- 默认管理员记录在 `docs/012_RBAC权限模型与种子数据_2026-07-05.md`。
- 默认 JWT secret 在 `config.yaml` 中明确标注为开发占位值。
- `validateConfig()` 在生产环境会拒绝不安全 JWT secret。

建议：

- 上线前必须替换 JWT secret。
- 上线前必须替换或禁用默认管理员。
- 如项目不希望自动 seed，可设置 `DB_SEED_DEFAULTS=false`。

## 9. 推荐的复用策略

### 9.1 第一个项目

建议保持脚手架完整，只新增业务资源。目标是验证：

- 认证能否满足业务。
- RBAC 粒度是否够用。
- 新增资源流程是否顺畅。
- 测试门禁是否能覆盖核心风险。

不要急着抽代码生成器。

### 9.2 第二到第三个项目

开始观察重复点：

- 是否每个项目都重复写相同 model/controller/routes 模板。
- 是否每个项目都重复写相同审计代码。
- 是否权限 seed 和菜单 seed 重复度很高。
- 是否配置项和部署脚本重复度很高。

只有当重复在多个真实项目中出现，再考虑抽 helper 或 scaffold 命令。

### 9.3 长期演进

优先级建议：

1. 稳定新增资源模板。
2. 提升测试数据隔离。
3. 增加 CI 中 SQLite 全量测试和 MySQL 测试库验证。
4. 在多版本生产升级前评估迁移工具。
5. 当 2-3 个项目验证模板稳定后，再考虑 `bun run scaffold:resource <name>`。

## 10. 结论

当前脚手架已经具备后台管理类项目复用价值。它的核心优势不是功能多，而是把后台项目最常重复、最容易写散的能力收敛成一个可启动、可测试、可扩展的底座。

正确使用方式是：

- 先跑通脚手架。
- 再按模板新增业务资源。
- 遇到重复再抽象。
- 所有变更同步文档和测试。
- 不为了“看起来更架构化”而制造大而全框架。

一句话：**把它当作后台项目的可靠起点，而不是一套必须不断膨胀的平台。**
