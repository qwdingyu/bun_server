### 1.8 半成品批量用户接口

状态：已完成。

证据：

- `src/routes/users.js` 已移除 `NOT_IMPLEMENTED` 返回。
- `src/controllers/UserController.js` 新增 `batchUpdateStatus()` 与 `batchDeleteUsers()`。
- `src/models/UserModel.js` 新增 `batchUpdateStatus()` 与 `batchSoftDeleteUsers()`。
- `tests/test-user-batch-routes.js` 验证批量静态路由不会被 `/:id` 抢匹配。

规则：

- 仅 `super_admin` 可访问。
- 不能操作当前登录用户自己。
- 批量删除使用软删除。

踩坑记录：

- 背景：`DELETE /batch` 与 `DELETE /:id` 同属用户路由。
- 原因：动态路由若先注册，容易抢先匹配静态批量路由。
- 排查过程：新增批量接口测试后发现响应状态不符合预期，逐步读取响应定位到路由和校验链路。
- 解决方案：将 `/batch/update-status` 与 `/batch` 放到 `/:id` 前面注册，并新增专项测试固定行为。
- 建议：后台脚手架中所有静态管理路由都应前置于动态参数路由。

附加修正：

- `src/middleware/validation/index.js` 中 JSON Schema `integer` 现在接受 JS `number` 且要求 `Number.isInteger()`，避免 JSON 整数被误判为类型错误。

### 3.1 全量测试可信度踩坑

- 背景：运行 `bun run tests/run-tests.js` 时最终显示“所有测试通过”。
- 原因：部分旧测试脚本内部记录失败但没有以非 0 状态退出，导致启动器误判。
- 排查过程：读取全量测试输出后发现 `test-api.js` 多处断言失败，但脚本进程仍被 `run-tests.js` 识别为成功。
- 解决方案：已按 `010_后台接口契约_2026-07-05.md` 重写 `tests/test-api.js`，只验证当前真实接口，并在失败时退出非 0。
- 建议：后续继续整理各测试文件的数据隔离策略，避免测试顺序和共享数据库状态互相影响。
