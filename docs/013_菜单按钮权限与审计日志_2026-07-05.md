# 菜单按钮权限与审计日志

> 目标：在不引入重型权限平台的前提下，为后台管理项目提供可复用、可裁剪、可测试的菜单/按钮门禁和关键操作审计闭环。

## 1. 设计结论

当前采用“小而完整”的第一版方案：

- 菜单元数据放在代码中维护，便于脚手架复制、裁剪和代码评审。
- 用户有效权限来自现有 RBAC 模型，不新增第二套权限来源。
- 前端通过 `/api/users/me/permissions` 初始化权限名数组。
- 前端通过 `/api/users/me/menus` 获取已过滤菜单和按钮。
- 审计日志写入 `audit_logs`，先覆盖用户管理高价值变更操作。

暂不做：数据库菜单表、可视化菜单编辑器、复杂数据权限、审计日志后台查询页。这些能力会显著增加维护成本，不适合作为复用脚手架默认能力。

## 2. 运行流程

### 2.1 登录后初始化

```text
1. 前端登录成功，保存 accessToken 和 refreshToken。
2. 前端保存登录响应中的 user，或调用 GET /api/users/me 获取最新用户。
3. 前端调用 GET /api/users/me/permissions 获取 permissionNames。
4. 前端调用 GET /api/users/me/menus 获取菜单树和按钮清单。
5. 前端根据 menus 渲染侧边栏，根据 permissionNames 做路由和按钮体验控制。
6. 后端接口继续用中间件和模型做真实权限校验。
```

### 2.2 菜单过滤

`src/modules/rbac/menu.js` 中的菜单节点只包含后台项目最常见的元数据：

```json
{
  "key": "users",
  "title": "用户管理",
  "path": "/users",
  "icon": "users",
  "permission": "users:read",
  "buttons": [
    { "key": "create", "title": "新增用户", "permission": "users:create" }
  ]
}
```

过滤规则：

- `permission` 为空的菜单默认可见，例如控制台。
- 菜单 `permission` 命中当前用户有效权限才返回。
- 按钮按自身 `permission` 独立过滤。
- 前端隐藏按钮只提升体验，不能作为安全边界。

### 2.3 审计写入

`AuditLogModel.record()` 负责写入审计日志：

- `user_id`：操作者。
- `action`：动作，例如 `users.batch_update_status`。
- `resource_type`：资源类型，例如 `users`。
- `resource_id`：资源 ID 或批量 ID 列表。
- `old_values/new_values`：JSON 字符串。
- `ip_address/user_agent`：请求来源。
- `status/error_message`：执行状态。

审计写入失败不阻断主流程，但会写入应用日志。这是后台脚手架的务实选择：关键业务操作不能因为审计库短暂异常全部失败，但生产项目如有强合规要求，可改为强一致写入。

## 3. 已覆盖范围

已接入审计的操作：

- 更新用户：`users.update`
- 删除用户：`users.delete`
- 切换用户状态：`users.toggle_status`
- 批量更新状态：`users.batch_update_status`
- 批量删除用户：`users.batch_delete`

已补齐的持久化映射：

- SQLite：`src/models/schema/sqlite.js`
- MySQL：`src/models/schema/mysql.js`
- 原始 SQLite DDL：`src/db/sqlite-schema.sql` 已存在 `audit_logs`

## 4. 扩展新后台资源的规则

新增业务资源时按以下顺序落地：

1. 在 seed 中增加权限点，命名遵循 `resource:action`。
2. 在 `src/modules/rbac/menu.js` 增加菜单和按钮元数据。
3. 在后端路由或控制器中继续做真实权限校验。
4. 对新增、更新、删除、批量、审批等高价值操作写入审计日志。
5. 增加接口测试，至少覆盖权限过滤和审计写入其中一个闭环。
6. 同步更新 `docs/010_后台接口契约_2026-07-05.md` 和前端集成文档。

## 5. 踩坑记录

### 5.1 权限名必须与种子数据一致

背景：菜单配置中一度使用 `system:config`，但默认 seed 中实际权限为 `config:read` 和 `config:update`。

原因：菜单元数据和 RBAC seed 如果分散维护，很容易出现权限名漂移。

排查过程：对照 `src/db/seed.js` 的 `DEFAULT_PERMISSIONS` 和菜单配置，发现系统配置菜单权限无法被任何默认角色命中。

解决方案：菜单配置改为使用 `config:read`，保存按钮使用 `config:update`。

建议：新增菜单前先从 seed 或权限管理模型确认权限点名称；后续如权限点增长，可增加菜单权限名一致性测试。

### 5.2 审计日志不应重复造第二套日志系统

背景：项目已有应用日志和业务日志，同时 SQLite DDL 中已有 `audit_logs` 表。

原因：如果审计继续只写文件日志，后台管理项目无法按用户、资源、动作查询关键变更；如果重新设计复杂日志平台，又会偏离轻量脚手架目标。

排查过程：确认 `sqlite-schema.sql` 已有 `audit_logs`，但 Drizzle schema 和模型未映射，控制器也未写入。

解决方案：新增 `AuditLogModel.record()`，复用现有表结构，先接入用户管理高价值变更操作。

建议：审计查询后台、导出、保留周期、归档策略按具体项目合规要求扩展，不进入默认脚手架核心。
